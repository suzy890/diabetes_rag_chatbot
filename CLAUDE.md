# CLAUDE.md — 코딩 에이전트 최상위 규칙

> **문서 버전:** v0.1 · **작성 기준일:** 2026-07-08
> 이 문서는 이 저장소에서 코드를 작성·수정하는 AI 코딩 에이전트가 **항상 지켜야 할 최상위 규칙**이다.
> 아래 규칙과 사용자의 직접 지시가 충돌하면 사용자의 지시를 우선하되, 규칙을 벗어나는 경우 먼저 알린다.
> 세부 규칙은 [.claude/rules/](.claude/rules/)의 architecture · research-data · medical-safety 문서를 함께 따른다.

---

## 1. 프로젝트 정의 (임의로 바꾸지 말 것)

- 이 프로젝트는 **고령 제2형 당뇨 환자를 대상으로 한 연구용 반응형 웹앱**이다.
- 목적은 **AI 모델 개발이 아니라 행동지원 시스템 구축**이다. 새로운 LLM 학습·fine-tuning·멀티에이전트 검증을 하지 않는다.
- 연구설계는 **실험군·대조군이 없는 단일군 현장 실증 연구**다. 따라서 **실험군·대조군 배정 기능을 만들지 않는다.**
- **네이티브 앱이 아니라 반응형 웹앱**으로 만든다. 스마트폰·태블릿·노트북에서 같은 주소로 접속한다.
- 최종 산출물은 웹앱뿐 아니라 **분석 가능한 연구 데이터**다. 그래서 화면보다 데이터 구조를 먼저 따른다.

## 2. 기술 규칙

**사용하는 것**
- 언어: **Python**
- 웹 화면: **Streamlit**
- 데이터베이스: **Supabase (PostgreSQL)**
- 벡터 검색: **Supabase pgvector**
- AI 응답: **외부 LLM API**
- 임베딩: **외부 Embedding API**
- 버전 관리: **GitHub**

**하지 않는 것 / 제한**
- 초기 버전에서 **LangChain 등 대형 프레임워크를 도입하지 않는다.** (요청받지 않은 프레임워크 추가 금지)
- **마이크로서비스가 아니라 모듈형 단일 애플리케이션**으로 구성한다.
- **핵심 실행 Python 코드는 500줄 이내**를 유지한다. (아래 6개 파일 기준. SQL 스키마, 프롬프트, 규칙 문서, 테스트 문서는 이 줄 수에서 제외)
- **API 키·비밀정보를 코드에 직접 작성하지 않는다.** 환경변수/시크릿으로 관리하고 저장소에 올리지 않는다([.gitignore](.gitignore) 참고).

## 3. 파일별 책임

핵심 실행 코드는 `src/` 아래 6개 모듈로 나눈다. 각 파일은 자기 책임 범위만 담당한다.

| 파일 | 책임 | 목표 줄 수 |
|------|------|-----------|
| `app.py` | 화면과 전체 사용자 흐름, 각 기능 호출 | 약 180~220 |
| `rag.py` | 검색 및 근거 기반 답변 | 약 90~110 |
| `nudge.py` | 넛지 트리거 조건과 템플릿 선택 | 약 60~80 |
| `database.py` | Supabase 저장·조회 | 약 60~80 |
| `safety.py` | 위험 질문 판별과 제한 응답 | 약 30~50 |
| `config.py` | 환경설정 로딩과 버전값 | 약 20 |

> UI(`app.py`)에서 LLM이나 벡터 DB를 **직접 호출하지 않는다.** 반드시 `rag.py`·`nudge.py`·`database.py` 등 담당 모듈을 거친다. (자세한 내용은 [.claude/rules/architecture.md](.claude/rules/architecture.md))

## 4. 상태관리 원칙

- `st.session_state`는 **화면의 임시 상태에만** 사용한다. (예: 지금 입력 중인 값, 화면 전환 상태)
- **연구 데이터는 Supabase에 영구 저장**한다. 화면 상태와 연구 데이터를 분리한다.
- 새로고침 이후에는 **DB에서 대화와 상태를 복원**한다. (session_state에만 의존하지 않는다)
- 모든 메시지와 이벤트에는 **고유 ID**를 부여한다.
- **Streamlit 재실행(rerun)으로 인한 중복 저장을 방지**한다. (같은 이벤트가 두 번 기록되지 않도록 멱등하게 처리)

## 5. 연구 데이터 · 안전 규칙 (요약 — 세부는 별도 문서)

- 모든 연구 이벤트에는 **참여자 ID · 발생 시각 · 시스템 버전**을 기록한다. (인증 전 이벤트만 참여자 ID 예외 — 세부는 규칙 문서) → [.claude/rules/research-data.md](.claude/rules/research-data.md)
- **진단·처방·복약 변경을 직접 지시하지 않는다.** 근거가 부족하면 답변을 보류한다. → [.claude/rules/medical-safety.md](.claude/rules/medical-safety.md)
- 실명·전화번호·주소·환자번호 등 **직접식별정보를 저장하거나 외부 API로 전송하지 않는다.**

## 6. 작업 방식 (코딩 에이전트 행동 규칙)

- **한 번에 하나의 작은 작업만** 수행한다. 큰 요청은 작은 단위로 쪼갠다.
- **요청하지 않은 기능을 임의로 추가하지 않는다.** 범위를 벗어난 아이디어는 제안만 하고 승인받은 뒤 진행한다.
- 아래 작업은 **실행 전에 반드시 사용자 승인**을 받는다:
  - 파일 삭제
  - 데이터베이스 스키마 변경
  - 배포
- 코드를 수정한 뒤에는 **테스트를 실행**한다.
- 수정한 **파일과 실행 결과를 보고**한다.
- **실패했을 때 성공했다고 보고하지 않는다.** 오류·미완료는 있는 그대로 알린다.
- 범위·구조·규칙을 바꿔야 할 상황이면, 임의로 바꾸지 말고 먼저 사용자에게 확인한다. (해당 규칙 문서 / PRD를 먼저 갱신)

## 1. 대원칙

> **⚠️ 방향 전환 (2026-07-19 · DECISIONS D44/D45):** 연구팀 논의로 **페이퍼의 주 데이터원이 상호작용 로그가 아니라 사용 후 설문(사용 경험)** 으로 정해졌다. 그래서 우선순위가 **데이터 정밀도 < 완성도·친숙도·사용성** 으로 이동한다.
> - 아래 "과정 데이터가 최종 산출물"·측정 순수성 논리는 **여전히 로그를 남기는 근거로는 유효**하나, 더 이상 화면·세션 설계를 지배하지 않는다. 로그는 **최대한 확보하되 주 분석 대상은 설문**이다.
> - **2시간 세션 리셋은 폐기**하고 대화 히스토리를 지속 노출한다(D45). 이로 인한 반복질문 confound는 팀이 수용했다.
> - **로깅 필수 항목(참여자ID·시각·시스템 버전)과 개인정보 원칙(§2·§5)은 그대로 유지된다.**

- **최종 설문 점수만으로는 부족하다.** 행동 변화의 흐름을 분석하려면 과정 데이터(넛지 예정/노출/반응, 질문·검색·답변, 행동 약속·수행)가 함께 있어야 한다.
- 설계 순서는 **사용자 행동 흐름 → 저장할 이벤트 → 테이블·필드 → 코드**다. 화면부터 만들고 나중에 데이터를 붙이지 않는다.
- 연구 데이터는 **Supabase에 영구 저장**한다. 화면 임시 상태(`st.session_state`)와 분리한다.

## 2. 필수 기록 항목 (모든 연구 이벤트에 공통)

모든 연구 이벤트/기록에는 다음을 반드시 남긴다.

- **참여자 ID** (`participant_id`, 익명) — _단, 인증 전 이벤트는 예외(아래)_
- **발생 시각** (timestamp) — 종단분석에서 순서 재구성에 필수 · **예외 없음**
- **시스템 버전** (`system_version_id`) — 어떤 설정으로 중재했는지 추적 · **예외 없음**

> **인증 전 이벤트 예외 (2026-07-13 결정):** `app_opened`처럼 참여자가 코드를 입력하기 **전**에 발생하는 이벤트는 참여자·세션을 알 수 없으므로 `participant_id`·`session_id`가 비어 있을 수 있다.
> **인증 이후의 모든 이벤트는 반드시 참여자 ID가 채워진다.** 발생 시각과 시스템 버전은 어떤 경우에도 필수다.
> (이 예외로 "앱은 열었지만 코드를 넣지 않고 이탈"한 경우를 기록해 접근성·이탈 분석에 쓸 수 있다.)

> 연구 중간에 설정(모델·프롬프트·넛지 규칙·지식베이스)이 바뀌면 참여자마다 다른 중재를 받은 셈이 되므로, 모든 데이터에 시스템 버전이 연결되어야 한다.

## 3. participant · session · message · event 구분

네 층위를 명확히 구분해 기록한다. (혼용 금지)

| 구분 | 의미 | 예 |
|------|------|-----|
| **participant** | 연구 참여자 1명 | `P001` |
| **session** | 한 번의 접속(시작~종료) | 오늘 오후 접속 |
| **message** | 채팅창에 나타난 한 건의 메시지 | 사용자 질문 / AI 답변 / 넛지 |
| **event** | 화면 열기·클릭·행동 선택 등 시간순 사건 | `nudge_displayed`, `source_clicked` |

- **message는 `role`(user/assistant/system)과 `message_type`(nudge / nudge_response / rag_question / rag_answer / safety_message 등)을 분리**해 기록한다. AI가 보낸 메시지라도 넛지인지 RAG 답변인지 안전 안내인지 연구적으로 의미가 다르다.
- **행동의도(하겠다)와 실제 수행(했다)을 분리**해 기록한다. (넛지의 `action_commitment` ↔ 추후 확인의 `completion_status`)
- **예정 시각과 실제 노출 시각을 분리**해 기록한다. (넛지 `scheduled_at` ↔ `displayed_at`) 사용자가 앱을 열지 않으면 예정은 됐어도 노출은 안 될 수 있다.
- **RAG는 검색 과정 자체를 저장**한다. 최종 답변만 저장하면 할루시네이션이 검색 실패인지 생성 실패인지 알 수 없다. (검색 청크·유사도 점수·선택 청크·top_k·지식베이스 버전)

## 4. 중복 저장 금지

- Streamlit은 상호작용마다 스크립트를 재실행(rerun)한다. **같은 이벤트·메시지가 두 번 저장되지 않도록** 멱등하게 처리한다.
- 메시지·이벤트·넛지에는 고유 ID를 부여하고, 새로고침/재실행 시 이미 저장된 기록을 다시 쓰지 않는다.
- 저장 성공 여부를 확인하고, 실패는 `technical_errors`에 남긴다. (기술 오류를 연구 행동 데이터와 분리해야, 사용자의 무응답이 실제 무관심인지 시스템 실패인지 구분할 수 있다)

## 5. 직접식별정보 저장 제한 (개인정보)

- **실명·전화번호·주소·환자번호 등 직접식별정보를 저장하지 않는다.** 참여자는 익명 `participant_id`로만 관리한다.
- 직접식별정보를 **외부 LLM/임베딩 API로 전송하지 않는다.** 외부 API에는 답변 생성에 필요한 최소한의 맥락만 보낸다.
- 실명과 `participant_id`를 잇는 **신원 연결표는 애플리케이션 데이터와 분리**해 별도로 보관한다.
- 자유서술 입력에 직접식별정보가 섞일 가능성을 고려해, 필요 시 입력 안내·제한·마스킹을 검토한다.
- 데이터 보관기간·접근권한·외부 API 이용·참여 철회 시 처리 범위는 **IRB / 연구윤리 기준**에 맞춰 별도 확정한다. _(미결정 — PRD 부록 B 참고)_

## 6. 분석을 위해 특히 놓치지 말 것

- 정확한 timestamp (모든 사건)
- 넛지 노출과 반응의 **연결 ID**, 행동 약속과 수행 확인의 **연결 ID**
- 반복 측정 시점 (baseline / week1 / week2 / post 등 — 시점 정의는 미결정)
- 외부 API의 토큰·비용·지연시간 (`model_calls`)
- 시스템 버전 (`system_versions`)

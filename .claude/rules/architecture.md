## 1. 전체 시스템 구조

초기 버전은 복잡한 프레임워크 없이 **세 층을 직접 연결**하는 단순한 구조다.

```
[ 반응형 웹앱 화면 (Streamlit / app.py) ]
              ↕
[ Supabase : PostgreSQL 데이터 저장 + pgvector 벡터 검색 ]
              ↕
[ 외부 API : LLM(답변·넛지 문장화) · Embedding(문서·질문 벡터화) ]
```

하나의 채팅 화면 안에서 **선제적 넛지**와 **RAG 질문·답변**이 함께 작동하며, 모든 상호작용은 연구 데이터로 저장된다.

## 2. 처리 흐름 (대화 오케스트레이터)

사용자 입력이 들어오면 **무엇인지 먼저 구분**한 뒤 경로를 나눈다.

```
입력 구분
 ① 예정된 넛지인가?
 ② 사용자의 일반 응답인가?
 ③ 당뇨 관련 질문인가?
 ④ 위험하거나 제한해야 하는 질문인가?
        │
        ├─▶ 넛지 처리 경로 (nudge.py)
        │     시간·상황 확인 → 규칙 선택 → 승인 템플릿 선택
        │     → 필요 시 LLM 문장화 → 응답 수집 → 추후 수행 확인
        │
        └─▶ RAG 처리 경로 (rag.py)
              질문 전처리 → 질문 임베딩 → 청크 검색
              → 근거 충분성 확인 → 근거 기반 답변 생성 → 출처 표시
        │
        ▼
   안전성 처리 (safety.py)  ── 응급·고위험 탐지 / 진단·처방·복약 제한 / 근거 부족 시 보류 / 고정 안내문
        │
        ▼
   연구 데이터 저장 (database.py) ── Supabase
```

## 3. 모듈별 책임

| 모듈 | 책임 | 하지 않는 것 |
|------|------|-------------|
| `app.py` | 화면 렌더링, 사용자 흐름, 각 모듈 호출(오케스트레이션) | LLM·벡터 DB 직접 호출 안 함 |
| `rag.py` | 질문 임베딩 요청, 청크 검색, 근거 기반 답변 생성, 출처 구성 | 화면 렌더링 안 함 |
| `nudge.py` | 넛지 트리거 판단, 규칙·템플릿 선택, 필요 시 LLM 문장화 요청 | 화면 렌더링 안 함 |
| `database.py` | Supabase 저장·조회 (참여자·세션·메시지·이벤트·넛지·검색·비용 등) | 비즈니스 판단 안 함 |
| `safety.py` | 위험 질문 판별, 제한 응답·고정 안내문 반환 | 진단·처방 생성 안 함 |
| `config.py` | 환경설정·키 로딩, 버전값 제공 | 비밀정보를 코드에 하드코딩 안 함 |

## 4. 핵심 구조 원칙

- **UI에서 직접 LLM 또는 벡터 DB를 호출하지 않는다.** `app.py`는 항상 담당 모듈(`rag.py`, `nudge.py`, `database.py`)을 거친다.
- 외부 API 호출은 담당 모듈 안에 감싼다. 모델·제공사가 바뀌어도 `app.py`는 영향받지 않도록 한다.
- 모든 외부 API 호출은 토큰·비용·지연시간을 기록한다(→ `database.py`, `model_calls`).
- 데이터 저장 책임은 `database.py`에 모은다. 다른 모듈이 SQL을 여기저기 흩뿌리지 않는다.

## 5. 기능 간 의존관계

```
app.py  ──▶ nudge.py ──▶ (LLM API)        [넛지 문장화, 필요 시만]
        ──▶ rag.py   ──▶ (Embedding API)  [질문 임베딩]
        │            ──▶ (Vector 검색)     [database.py 경유]
        │            ──▶ (LLM API)         [답변 생성]
        ──▶ safety.py                      [위험 질문 판별]
        └──▶ database.py ──▶ (Supabase)    [모든 저장·조회]
config.py ──▶ (모든 모듈에 설정·버전 제공)
```

- 의존은 **위에서 아래로만** 흐른다. `database.py`나 `config.py`가 `app.py`를 호출하지 않는다.

## 6. 향후 교체 가능한 구성요소

초기 단순 구조를 유지하되, 아래는 나중에 **교체·확장 가능**하도록 담당 모듈 뒤에 숨긴다.

- LLM 제공사·모델 (rag.py / nudge.py 내부)
- 임베딩 제공사·모델 (rag.py 내부)
- 벡터 검색 방식 (database.py 내부, 현재 pgvector)
- 넛지 트리거 (현재 접속 시점 → 향후 예약 넛지, 날씨 기반 등은 PRD 제외 범위)

> 교체 가능성을 위해 지금부터 대형 프레임워크를 미리 넣지 않는다. 필요할 때 담당 모듈만 바꾼다.

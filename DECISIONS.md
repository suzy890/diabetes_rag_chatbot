## 확정된 결정

| # | 결정 | 이유 |
|---|------|------|
| D1 | **Streamlit** 사용 | 비개발자가 채팅형 웹앱을 비교적 빠르게 구현 가능. 바이브코딩에 적합. |
| D2 | **Supabase** 사용 | 참여자·세션·설문·이벤트를 표로 저장하고, 같은 DB에서 벡터 검색까지 가능. |
| D3 | **PostgreSQL + pgvector** | 관계형 저장과 문서 임베딩 유사도 검색을 **하나의 DB**에서 처리(운영 단순화). |
| D4 | **네이티브 앱 제외**, 반응형 웹앱 | 소규모 연구 프로토타입. 앱스토어 배포·OS별 개발 부담 회피. 한 URL로 접근. |
| D5 | **외부 푸시 알림 제외**(초기) | 인증·동의·비용·알림 실패 처리 부담. MVP는 접속 시점 넛지로 충분. |
| D6 | **승인 템플릿 중심 넛지** | 참여자마다 다른 중재를 받으면 결과 해석이 어려움. LLM 자유생성 제한 → 일관성·비용. |
| D7 | **LangChain 등 초기 미도입** | 여러 도구를 동시에 쓰면 어떤 코드가 무슨 일을 하는지 파악이 어려움. 세 층 직접 연결이 단순·명확. |
| D8 | **모듈형 단일 애플리케이션** | 마이크로서비스는 과함. 6개 모듈로 나눈 단일 앱이 이해·유지에 유리. |
| D9 | **연구 데이터와 화면 상태 분리** | 최종 산출물이 분석 가능한 데이터. session_state(임시)와 Supabase(영구)를 분리해 새로고침에도 데이터 보존. |
| D10 | **핵심 실행 코드 500줄 제한** | 단순함 유지. 문서·프롬프트·SQL·테스트는 제외해 규칙을 읽고 수정하기 쉽게. |
| D11 | **수집 우선(collect-now, analyze-later)** | (연구팀 피드백) 이번에 분석 안 해도 추후 가치 있는 데이터는 지금 최소 수집. 단 개인정보 동의 부담 항목은 제외. → [DATA_SCHEMA.md](DATA_SCHEMA.md) |
| D12 | **역순 접근으로 결정 정리** | (연구팀 피드백) 실증 환경·최소 결과물부터 정의하고 거기서 미결정 항목을 도출. → [docs/analysis-targets.md](docs/analysis-targets.md) |
| D13 | **개발 기록을 노션에 인터랙티브 HTML로 병행** | 비개발자 공동연구자가 진행을 쉽게 이해하도록. 위젯 원본은 `docs/widgets/`에서 버전관리. |
| D14 | **인증 전 이벤트 기록 허용** (`events.participant_id` nullable) | `app_opened`는 코드 입력 전에도 발생 → 참여자를 알 수 없음. "앱은 열었지만 코드 미입력 이탈"을 기록해 고령 사용자 **접근성·이탈 분석**에 활용. 인증 후 이벤트는 항상 참여자 ID를 채운다. |
| D15 | **`retrieval_logs` 정규화 + `retrieval_chunks` 신설** | (연구팀 리뷰) 검색 청크를 배열로 담으면 "청크별 검색 빈도·유사도 분포·선택률" 분석이 사실상 불가능. 검색 1회=1행 / 청크 1개=1행으로 분리해야 RAG_RULES의 "검색·답변 품질 분리 평가"가 실제로 가능. |
| D16 | **`sessions.app_version` 컬럼 제거** | (연구팀 리뷰) 웹앱은 "기기에 설치된 앱 버전"이 없다(모두 같은 서버). `system_version_id`로 알 수 있으므로 순수 중복이며 불일치 위험만 생김. |
| D17 | **`events` 참여자·세션 일관성을 복합 FK로 DB가 보장** | (연구팀 리뷰) "P001 세션인데 이벤트는 P002" 같은 오염을 앱 코드 규율에 맡기지 않고 **DB가 물리적으로 차단**. NULL이면 검사 생략되어 인증 전 이벤트와도 양립. |
| D18 | **RLS 켜고 정책 0개 + service_role 키(서버 전용)** | Streamlit은 100% 서버에서 실행되어 키가 브라우저로 나가지 않는다. 외부는 읽기·쓰기 전면 차단 → 주소를 알아도 **가짜 연구 데이터를 넣을 수 없다**. |
| D19 | **500줄 제한 = 빈줄·주석 제외한 실제 코드** | 규칙의 목적은 시스템 단순화이지 설명 축소가 아니다. 비개발자 연구자가 코드를 읽을 수 있도록 한국어 주석은 충분히 남긴다. 확인: `tests/check_line_limit.py` |
| D20 | **넛지 영역=식사(간식·야식 포함) · 하루 3회 상한 · 템플릿 v0.1** | (연구팀 확정 2026-07-13) 혈당 변동에 직결되는 섭식 행동을 먼저 다룬다. 야식 문구는 **훈계·죄책감 없이 사실만 묻는다**(안전 규칙). |
| D21 | **데모 배포: Streamlit Community Cloud · 비공개** | 비개발자 공동연구자가 클론·설치 없이 **URL 하나로** 진행 상황을 보고 피드백할 수 있다. 무료이고 GitHub push 시 자동 재배포. **비공개(이메일 허용목록)** 로 외부인의 가짜 데이터 유입을 막는다. → [DEPLOY.md](DEPLOY.md) |
| D22 | **연구자용 관리자 대시보드를 MVP 범위에 추가** (2026-07-13) | 연구자 혼자면 Supabase 대시보드로 충분하지만, ① **공동연구자가 비개발자**라 원시 테이블을 볼 수 없고, ② **학교·기관과 결과를 공유**하려면 시각화된 요약이 필요하다. 즉 "있으면 좋은 기능"이 아니라 **연구 운영상 실제 필요**다. → [PRD.md](PRD.md) §8-1, [TASKS.md](TASKS.md) T4.9 |
| D23 | **관리자 도구는 500줄 제한 대상 아님 · 별도 앱으로 분리** | 500줄 제한의 목적은 **참여자용 핵심 실행 코드**(app·rag·nudge·database·safety·config)를 단순하게 유지하는 것이다. 연구자용 관리 도구는 성격·사용자·배포가 모두 다르므로 `admin/`으로 분리한다. **보안상으로도 분리가 필수** — 관리자 화면은 전체 참여자 데이터를 보므로 참여자용 앱과 같은 주소에 두면 안 된다. |
| D24 | **관리자 대시보드는 연구 행동 데이터에 대해 읽기 전용** | 수정·삭제 기능을 두면 실수나 오조작으로 **연구 데이터 무결성**이 깨질 수 있다. 참여자 등록·상태 변경만 허용하고, messages·events·nudge_events·retrieval_logs는 읽기만 가능하게 한다. |

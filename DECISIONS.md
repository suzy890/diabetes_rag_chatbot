## 확정된 결정

| # | 결정 | 이유 |
|---|------|------|
| D1 | **Streamlit** 사용 | 비개발자가 채팅형 웹앱을 비교적 빠르게 구현 가능. 바이브코딩에 적합. |
| D2 | **Supabase** 사용 | 참여자·세션·설문·이벤트를 표로 저장하고, 같은 DB에서 벡터 검색까지 가능. |
| D3 | **PostgreSQL + pgvector** | 관계형 저장과 문서 임베딩 유사도 검색을 **하나의 DB**에서 처리(운영 단순화). |
| D4 | **네이티브 앱 제외**, 반응형 웹앱 | 소규모 연구 프로토타입. 앱스토어 배포·OS별 개발 부담 회피. 한 URL로 접근. |
| D5 | **외부 푸시 알림 제외**(초기) | 인증·동의·비용·알림 실패 처리 부담. MVP는 접속 시점 넛지로 충분. |
| D6 | **승인 템플릿 중심 넛지** | 참여자마다 다른 중재를 받으면 결과 해석이 어려움. LLM 자유생성 제한 → 일관성·비용. |
| D7 | **LangChain 등 초기 미도입** | 여러 도구를 동시에 쓰면 어떤 코드가 무슨 일을 하는지 파악이 어려움. 세 층 직접 연결이 단순·명확. |
| D8 | **모듈형 단일 애플리케이션** | 마이크로서비스는 과함. 6개 모듈로 나눈 단일 앱이 이해·유지에 유리. |
| D9 | **연구 데이터와 화면 상태 분리** | 최종 산출물이 분석 가능한 데이터. session_state(임시)와 Supabase(영구)를 분리해 새로고침에도 데이터 보존. |
| D10 | **핵심 실행 코드 500줄 제한** | 단순함 유지. 문서·프롬프트·SQL·테스트는 제외해 규칙을 읽고 수정하기 쉽게. |
| D11 | **수집 우선(collect-now, analyze-later)** | (연구팀 피드백) 이번에 분석 안 해도 추후 가치 있는 데이터는 지금 최소 수집. 단 개인정보 동의 부담 항목은 제외. → [DATA_SCHEMA.md](DATA_SCHEMA.md) |
| D12 | **역순 접근으로 결정 정리** | (연구팀 피드백) 실증 환경·최소 결과물부터 정의하고 거기서 미결정 항목을 도출. → [docs/analysis-targets.md](docs/analysis-targets.md) |
| D13 | **개발 기록을 노션에 인터랙티브 HTML로 병행** | 비개발자 공동연구자가 진행을 쉽게 이해하도록. 위젯 원본은 `docs/widgets/`에서 버전관리. |
| D14 | **인증 전 이벤트 기록 허용** (`events.participant_id` nullable) | `app_opened`는 코드 입력 전에도 발생 → 참여자를 알 수 없음. "앱은 열었지만 코드 미입력 이탈"을 기록해 고령 사용자 **접근성·이탈 분석**에 활용. 인증 후 이벤트는 항상 참여자 ID를 채운다. |
| D15 | **`retrieval_logs` 정규화 + `retrieval_chunks` 신설** | (연구팀 리뷰) 검색 청크를 배열로 담으면 "청크별 검색 빈도·유사도 분포·선택률" 분석이 사실상 불가능. 검색 1회=1행 / 청크 1개=1행으로 분리해야 RAG_RULES의 "검색·답변 품질 분리 평가"가 실제로 가능. |
| D16 | **`sessions.app_version` 컬럼 제거** | (연구팀 리뷰) 웹앱은 "기기에 설치된 앱 버전"이 없다(모두 같은 서버). `system_version_id`로 알 수 있으므로 순수 중복이며 불일치 위험만 생김. |
| D17 | **`events` 참여자·세션 일관성을 복합 FK로 DB가 보장** | (연구팀 리뷰) "P001 세션인데 이벤트는 P002" 같은 오염을 앱 코드 규율에 맡기지 않고 **DB가 물리적으로 차단**. NULL이면 검사 생략되어 인증 전 이벤트와도 양립. |
| D18 | **RLS 켜고 정책 0개 + service_role 키(서버 전용)** | Streamlit은 100% 서버에서 실행되어 키가 브라우저로 나가지 않는다. 외부는 읽기·쓰기 전면 차단 → 주소를 알아도 **가짜 연구 데이터를 넣을 수 없다**. |
| D19 | **500줄 제한 = 빈줄·주석 제외한 실제 코드** | 규칙의 목적은 시스템 단순화이지 설명 축소가 아니다. 비개발자 연구자가 코드를 읽을 수 있도록 한국어 주석은 충분히 남긴다. 확인: `tests/check_line_limit.py` |
| D20 | **넛지 영역=식사(간식·야식 포함) · 하루 3회 상한 · 템플릿 v0.1** | (연구팀 확정 2026-07-13) 혈당 변동에 직결되는 섭식 행동을 먼저 다룬다. 야식 문구는 **훈계·죄책감 없이 사실만 묻는다**(안전 규칙). |
| D21 | **데모 배포: Streamlit Community Cloud · 비공개** | 비개발자 공동연구자가 클론·설치 없이 **URL 하나로** 진행 상황을 보고 피드백할 수 있다. 무료이고 GitHub push 시 자동 재배포. **비공개(이메일 허용목록)** 로 외부인의 가짜 데이터 유입을 막는다. → [DEPLOY.md](DEPLOY.md) |
| D22 | **연구자용 관리자 대시보드를 MVP 범위에 추가** (2026-07-13) | 연구자 혼자면 Supabase 대시보드로 충분하지만, ① **공동연구자가 비개발자**라 원시 테이블을 볼 수 없고, ② **학교·기관과 결과를 공유**하려면 시각화된 요약이 필요하다. 즉 "있으면 좋은 기능"이 아니라 **연구 운영상 실제 필요**다. → [PRD.md](PRD.md) §8-1, [TASKS.md](TASKS.md) T4.9 |
| D23 | **관리자 도구는 500줄 제한 대상 아님 · 별도 앱으로 분리** | 500줄 제한의 목적은 **참여자용 핵심 실행 코드**(app·rag·nudge·database·safety·config)를 단순하게 유지하는 것이다. 연구자용 관리 도구는 성격·사용자·배포가 모두 다르므로 `admin/`으로 분리한다. **보안상으로도 분리가 필수** — 관리자 화면은 전체 참여자 데이터를 보므로 참여자용 앱과 같은 주소에 두면 안 된다. |
| D24 | **관리자 대시보드는 연구 행동 데이터에 대해 읽기 전용** | 수정·삭제 기능을 두면 실수나 오조작으로 **연구 데이터 무결성**이 깨질 수 있다. 참여자 등록·상태 변경만 허용하고, messages·events·nudge_events·retrieval_logs는 읽기만 가능하게 한다. |
| D25 | **GraphRAG 미채택 — 일반 RAG + 하이브리드 검색** | GraphRAG의 강점은 **문서 간 종합 추론**인데, 그것은 우리가 **의도적으로 금지한 행위**다(개인 맞춤 의료 판단 금지). 또 근거가 "LLM이 생성한 커뮤니티 요약"이 되어 **출처 추적이 흐려지고**, 승인 문서 체계가 무너진다. 인덱싱 비용·복잡도도 예산·500줄 제한과 충돌. **참여자 질문은 단발성 사실 조회형**이라 일반 RAG가 적합. → 필요 여부는 `retrieval_chunks` 지표(유사도 분포·보류율·선택률)로 **데이터를 보고 판단**한다. |
| D26 | **용어 모호성은 '추론'이 아니라 '되묻기'로 처리** | 고령 참여자는 **"혈당"** 하나로 공복혈당·식후혈당·**당화혈색소**를 모두 가리킨다. 그런데 "혈당 7"과 "당화혈색소 7%"는 **정반대로 해석**된다. 시스템이 추측하면 관리가 안 되는 참여자를 "괜찮다"고 안심시킬 수 있다 → **안전 문제**. 모호성이 답을 바꿀 때만 되묻고, "모르겠다"를 허용한다. → [RAG_RULES.md](RAG_RULES.md) §3-1 |
| D27 | **승인 용어 사전에 임상 수치·기준을 넣지 않는다** | 사전은 "이 말이 무엇을 가리키는가"만 매핑한다. 수치·목표값을 사전에 박으면 **출처 없는 답변**이 되고, 문서가 개정돼도 낡은 값을 계속 말한다. 수치는 **승인 문서에서 검색해서** 답한다(근거 기반 원칙 유지). → [prompts/term-aliases.md](prompts/term-aliases.md) |
| D28 | **되묻기는 서비스 기능이 아니라 연구 관측 장치** | "참여자가 혈당과 당화혈색소를 구분하는가", "모른다고 답하는 비율은" 은 **AI 건강문해력의 직접 관측치**다. 설문보다 강력한 증거이며, **설계에 넣지 않으면 영영 관측할 수 없다**(데이터는 사후 소급 불가). `clarification_asked`/`clarification_answered` 이벤트로 기록. |
| D29 | **LLM·임베딩 제공사 = NVIDIA NIM (키 1개)** (2026-07-14) | 연구팀이 제공한 계정. `integrate.api.nvidia.com`은 **OpenAI 호환 엔드포인트**라 표준 방식으로 호출된다. **같은 키로 LLM과 임베딩을 모두 쓸 수 있어** 제공사·계정·비용 관리가 하나로 단순해진다(당초 우려했던 "임베딩 별도 제공사" 문제 해소). 실제 호출로 검증 완료 → [tests/check_llm_provider.py](tests/check_llm_provider.py) |
| D30 | **연구용 호출은 `temperature`를 낮게 고정** (2026-07-14) | 제공받은 예시 코드는 `temperature=1`(창의적 샘플링)이었다. 그러면 **같은 질문에 참여자마다 다른 답**이 나와 ① 중재가 표준화되지 않고 ② 재현성이 무너진다. 넛지를 승인 템플릿으로 고정한 이유(D6)와 동일한 논리. 근거 기반 답변에 필요한 것은 창의성이 아니라 **일관성**이다. |
| D31 | **시스템 프롬프트에서 '검색된 근거에 없는 수치'를 금지** (2026-07-14) | 검증 중 모델이 묻지도 않은 **임상 수치("6.5% 이상이면 당뇨병")를 스스로 생성**했다. 정확한 값이었지만 **출처 없는 답변**이며, 이는 D27(사전에 수치 금지)이 막으려던 문제와 같다. 근거 문서에 없는 수치·기준은 말하지 않도록 프롬프트에 명시한다. |
| D32 | **임베딩 모델 = `nvidia/llama-nemotron-embed-1b-v2` (2048차원)** (2026-07-14) | 사용 가능한 3종을 **고령 구어체 질문 10개로 실제 검색**시켜 비교했다 → [tests/check_embedding_models.py](tests/check_embedding_models.py). 결과: nemotron-embed **Top-1 100%**, nv-embed-v1 90%(4096차원·4.7배 느림), **nv-embedqa-e5-v5 10%**. ⚠️ **당초 "첫 후보"였던 e5-v5는 한국어를 사실상 이해하지 못한다**(1등·2등 점수차 0.005 = 모든 문서가 똑같아 보임). 같은 코드로 **영어 질문은 정상 동작**해 코드 문제가 아닌 **모델의 한국어 미지원**임을 확인했다. → `document_chunks`의 벡터 컬럼은 **`vector(2048)`** 으로 만든다. |
| D33 | **bge-m3 대신 nemotron 임베딩 확정 — 실제 문서로 재검증** (2026-07-15) | 연구팀이 문서를 bge-m3(1024차원)로 미리 임베딩해 와, nemotron과 비교 검토했다. bge-m3는 한국어 검색이 강하지만 **API가 아닌 로컬 2GB 모델**이라 질문 실시간 임베딩·Streamlit 배포에 부담(메모리)이 크다. nemotron을 **실제 승인문서 139청크 + 고령 구어체 질문 12개**로 재검증 → [tests/check_nemotron_realdocs.py](tests/check_nemotron_realdocs.py): **Top-1 58% · Recall@3 75% · Recall@5 92%**. Top-1이 낮은 원인은 **모델의 한국어 문제가 아니라 청크 잡음**(머리말·쪽번호·목차·참고문헌이 본문에 섞여 그 청크가 1등을 차지)임을 오답 분석으로 확인. → 임베딩은 nemotron 확정, **다음 레버는 청크 정제(T2.3)·크기 선택·하이브리드 검색(T2.9)**. |
| D34 | **Phase 2 RAG 테이블 5종 적용** (2026-07-15) | 마이그레이션 `phase2_rag_tables`로 `documents`·`document_chunks`(vector 2048)·`model_calls`·`retrieval_logs`·`retrieval_chunks` 생성. 기존 7테이블과 동일하게 **RLS 켜고 정책 없음**. **pgvector ANN 인덱스는 2000차원까지만** 지원해 2048 벡터엔 벡터 인덱스를 만들지 않음(전수 코사인 검색 = 정확도 손실 없음, 현재 소규모 코퍼스엔 충분). 규모가 커지면 `halfvec(2048)` 캐스팅으로 HNSW 가능. `schema.sql` 동기화 완료. |

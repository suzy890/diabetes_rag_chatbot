## 확정된 결정

| # | 결정 | 이유 |
|---|------|------|
| D1 | **Streamlit** 사용 | 비개발자가 채팅형 웹앱을 비교적 빠르게 구현 가능. 바이브코딩에 적합. |
| D2 | **Supabase** 사용 | 참여자·세션·설문·이벤트를 표로 저장하고, 같은 DB에서 벡터 검색까지 가능. |
| D3 | **PostgreSQL + pgvector** | 관계형 저장과 문서 임베딩 유사도 검색을 **하나의 DB**에서 처리(운영 단순화). |
| D4 | **네이티브 앱 제외**, 반응형 웹앱 | 소규모 연구 프로토타입. 앱스토어 배포·OS별 개발 부담 회피. 한 URL로 접근. |
| D5 | **외부 푸시 알림 제외**(초기) | 인증·동의·비용·알림 실패 처리 부담. MVP는 접속 시점 넛지로 충분. |
| D6 | **승인 템플릿 중심 넛지** | 참여자마다 다른 중재를 받으면 결과 해석이 어려움. LLM 자유생성 제한 → 일관성·비용. |
| D7 | **LangChain 등 초기 미도입** | 여러 도구를 동시에 쓰면 어떤 코드가 무슨 일을 하는지 파악이 어려움. 세 층 직접 연결이 단순·명확. |
| D8 | **모듈형 단일 애플리케이션** | 마이크로서비스는 과함. 6개 모듈로 나눈 단일 앱이 이해·유지에 유리. |
| D9 | **연구 데이터와 화면 상태 분리** | 최종 산출물이 분석 가능한 데이터. session_state(임시)와 Supabase(영구)를 분리해 새로고침에도 데이터 보존. |
| D10 | **핵심 실행 코드 500줄 제한** | 단순함 유지. 문서·프롬프트·SQL·테스트는 제외해 규칙을 읽고 수정하기 쉽게. |
| D11 | **수집 우선(collect-now, analyze-later)** | (연구팀 피드백) 이번에 분석 안 해도 추후 가치 있는 데이터는 지금 최소 수집. 단 개인정보 동의 부담 항목은 제외. → [DATA_SCHEMA.md](DATA_SCHEMA.md) |
| D12 | **역순 접근으로 결정 정리** | (연구팀 피드백) 실증 환경·최소 결과물부터 정의하고 거기서 미결정 항목을 도출. → [docs/analysis-targets.md](docs/analysis-targets.md) |
| D13 | **개발 기록을 노션에 인터랙티브 HTML로 병행** | 비개발자 공동연구자가 진행을 쉽게 이해하도록. 위젯 원본은 `docs/widgets/`에서 버전관리. |

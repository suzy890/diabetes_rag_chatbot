## 3. Phase 1 — 웹앱 · DB 골격 (07-13)

> **개발 원칙 — "껍데기(shell)"를 만들지 않는다.**
> 버튼만 있고 뒤가 빈 화면을 여러 개 만드는 대신, **하나의 기능을 데이터→로직→화면까지 관통(수직 슬라이스)** 시켜 완성한다.
> "완성"의 판정은 **화면이 아니라 데이터**로 한다 — 버튼이 눌리는가가 아니라, **올바른 데이터가 정확히 1건 쌓이는가**.

### Step 0 — 환경 준비
- Python **3.12** (기존 3.9는 지원 종료) · 가상환경 · `requirements.txt` 버전 고정
- 라이브러리 **4개만**: `streamlit` · `supabase` · `python-dotenv` · `watchdog`
- Supabase 프로젝트 `diabetes_rag_chatbot` (**서울 리전**) 생성
- 비밀정보는 `.env` / 배포 시크릿에만 — **코드·저장소에 절대 없음**

### Step 1 — DB 연결
- `src/config.py` (설정·버전) · `src/database.py` (Supabase 전담)
- **화면 0줄** — 배관만 연결하고 실제 연결을 검증

### Step 2 — 스키마 생성
**7개 테이블 생성** (설계 14개 중 지금 쓰는 것만 — 안 쓰는 테이블을 미리 만들면 그게 껍데기)

`system_versions` · `participants` · `sessions` · `events` · `messages` · `nudge_events` · `technical_errors`

**보안:** 전 테이블 **RLS 활성화 + 정책 0개** → 외부는 읽기·쓰기 전면 차단. 앱 서버(secret 키)만 기록 가능.
→ **주소를 알아도 가짜 연구 데이터를 넣을 수 없다.**

### T1.1 — 참여자 코드 입력 · 세션 생성 ✅
- 유효 코드(`P001`) → 세션 생성 + `session_started` 이벤트
- 미등록(`P999`) · 미활성(`P003`) → 차단, **DB에 아무것도 안 남김**
- ⭐ **새로고침 중복 방지** — 2겹 (`session_state` + DB에서 열린 세션 이어받기)
- 저장 실패 → `technical_errors` 기록
- **테스트 8/8 통과** + 실제 DB 데이터로 교차 검증

### T1.3 — 기본 채팅 화면 ✅
- 대화를 **화면 상태가 아니라 DB에서 읽음** → 새로고침해도 복원
- `role`(누가) × `message_type`(연구적 성격) 분리 저장
- 잘못된 값은 **DB의 CHECK 제약이 거부**
- **테스트 7/7 통과**

### T1.4 — 시간대 넛지: AI가 먼저 말을 검 ✅
**승인 템플릿 v0.1 (연구팀 승인)**

| 시간대 | 문구 | 응답 선택지 |
|---|---|---|
| 06–10시 | 안녕하세요. 오늘 아침 식사는 하셨어요? | 네, 먹었어요 / 아직이요 / 나중에 |
| 11–14시 | 점심 식사는 하셨어요? | 네, 먹었어요 / 아직이요 / 나중에 |
| **14–17시** | **오후에 간식 드셨어요?** | 네, 먹었어요 / 아니요 / 나중에 |
| 17–20시 | 저녁 식사는 하셨어요? | 네, 먹었어요 / 아직이요 / 나중에 |
| **21시 이후** | **저녁 드신 뒤에 뭔가 더 드셨어요?** | 네, 먹었어요 / 아니요 / 나중에 |

- **반복 제한:** 같은 넛지 하루 1회 · **하루 최대 3회** · 새로고침 중복 방지
- **안전:** 야식 넛지에 **훈계·죄책감 표현 금지** — 판단하지 않고 사실만 묻는다. 모든 넛지에 **"나중에 답할게요"**(거절할 권리) 제공
- **MVP는 LLM 미사용** — 템플릿 그대로 (일관성 + 비용 0원)
- **테스트 10/10 통과** (규칙 자체를 검증: "재노출을 실제로 막는가")

## 6. 연구팀 피드백 반영 내역

이 프로젝트의 설계는 **연구팀 피드백으로 실제로 개선**되었다.

| 피드백 | 반영 |
|---|---|
| "이번에 분석 안 해도 추후 위해 데이터를 모을 최소 스키마가 있으면" | ✅ **수집 우선(collect-now) 원칙** 도입 + `nudge_events.context_json`(미래 상황기반 넛지용). 단 위치 등 동의 부담 항목은 IRB 동의 전 미수집 |
| "역순으로, 최소 결과물부터 정의하고 결정사항을 정리하자" | ✅ **docs/analysis-targets.md** 신설 — 결과물 → 필요 데이터 → 확정할 결정 순으로 도출 |
| "답변 보류·안전 안내가 실증 때 과하게 뜨면 안 된다" | ✅ RAG **근거 충분성 3단계**(부분 근거면 일반정보+한계 명시), 안전 **5범주**로 고위험만 좁게 |
| ⭐ "검색 기록을 배열이 아니라 정규화해야 분석 가능" | ✅ **retrieval_logs 정규화 + retrieval_chunks 신설** — 청크별 검색빈도·유사도 분포·선택률을 SQL 한 줄로 |
| "sessions.app_version이 중복 아닌가" | ✅ **컬럼 제거** — 웹앱은 기기별 앱 버전이 없으므로 순수 중복 |
| "events.session_id 인덱스 확인" | ✅ **누락 확인 → 추가** |
| "간식(14–17시)·야식 넛지 추가하면?" | ✅ **템플릿 5종으로 확장** (야식은 죄책감 없는 문구로) |
| "관리자 페이지가 필요하다 (공동연구자 비개발자 + 학교 공유)" | ✅ **T4.9로 MVP 범위 추가** |

## 7. 개발 중 발견해 수정한 문제 (정직 기록)

**문서 검수만으로는 못 잡고, 코드로 옮기면서 드러난 모순들.** 이것이 v0.1 문서를 계속 갱신하는 이유다.

| # | 발견 | 조치 |
|---|---|---|
| 1 | **문서 상호 모순** — EVENT_DICTIONARY는 `app_opened`를 "인증 이전 포함"으로 정의했으나, DATA_SCHEMA는 `events.participant_id`를 필수로 규정. **코드를 짜려니 둘 다 만족 불가.** | 연구팀 결정 → **인증 전 이벤트 기록 허용**(nullable). 덕분에 "앱은 열었지만 코드 미입력 이탈"을 분석 가능. 3개 문서에 예외 명시 |
| 2 | **규칙을 지키려니 테이블이 하나 더 필요** — "모든 이벤트에 시스템 버전 기록" 규칙을 지키려면 `system_versions` 테이블이 있어야 함 | 3개 → **4개 테이블**로 시작 |
| 3 | **500줄 제한의 기준이 모호** — 전체 574줄 vs 실제 코드 332줄 | 연구자 결정 → **"주석·빈줄 제외한 실제 코드"** 로 확정. 규칙의 목적은 단순화이지 설명 축소가 아니므로 **한국어 주석은 유지**. 측정 도구(`check_line_limit.py`) 제작 |
| 4 | 앱 오류 "대화를 불러오지 못했습니다" | **`technical_errors` 테이블에 남은 로그로 원인 특정** — 코드 버그가 아니라 서버가 옛 모듈을 붙들고 있던 것. watchdog 설치로 해결. *(승인받아 만든 오류 로그 테이블이 30분 만에 값을 함)* |
| 5 | 테스트 자체의 오류 2건 | "안녕**하세요**"가 명령형 금지어에 걸린 오탐 / 행 수를 세는 대신 **규칙이 실제로 막는지**를 검증하도록 테스트 재작성 |

| 코드 줄 수 | **404 / 500** (여유 96줄) |
| 문서 일관성 | 10항목 통과 (Phase 0 종료 시점) |
| 배포 | HTTP 200 · 비공개 로그인 리다이렉트 확인 |

# 설계 노트 — 왜 이렇게 설계했는가

**고령 당뇨 환자를 위한 넛지 · RAG 웹앱**

> **작성일:** 2026-07-13 · **대상:** Phase 0·1 설계 결정
> 이 문서는 **"무엇을 만들었는가"가 아니라 "왜 그렇게 만들었는가"**를 기록한다.

---

## 이 문서의 위치

| 문서 | 성격 | 용도 |
|------|------|------|
| [DECISIONS.md](../DECISIONS.md) | **결정 로그** — 무엇을 정했나 (한 줄 요약 24건) | 개발 중 참조, 번복 방지 |
| **이 문서** | **설계 논거** — 왜 그렇게 정했나 (서술) | **논문 Methods 초고** · 설계 근거 · 포트폴리오 |

각 항목은 다음 다섯 가지로 구성한다.

> **결정** → **해결하려는 문제** → **이 설계가 없으면** → **대안과 트레이드오프** → **연구적 의미**

**핵심은 세 번째다.** "이 설계가 없으면 연구가 어떻게 망가지는가"를 말할 수 없다면, 그 설계는 정당화되지 않는다.

## 시작하기 전 — 정직한 전제 두 가지

**1. 이 설계는 혼자 만든 것이 아니다.**
아래 항목 중 **8번(검색 기록 정규화)**은 연구팀 리뷰가 기존 설계의 약점을 정확히 지적해 나온 것이고, **수집 우선 원칙**·**역순 접근**·**간식/야식 넛지**·**관리자 대시보드**도 연구팀 제안이다. 설계 품질은 개인이 아니라 **리뷰 루프**가 만들었다. 그 사실 자체가 기록될 가치가 있다.

**2. 이 설계는 아직 검증되지 않았다.**
설계가 우아한 것과 연구가 성립하는 것은 다르다. **진짜 시험은 "실증이 끝난 뒤 이 데이터로 연구 질문에 실제로 답할 수 있는가"**이다. `system_versions`가 정교해도 정작 분석 시점에 필요한 값이 비어 있으면 무용지물이다. 그래서 [analysis-targets.md](analysis-targets.md)(역순 접근 — 결과물부터 정의)가 함께 있어야 한다.

# 1. 화면보다 데이터 구조를 먼저 설계했다

**결정**
Phase 0(5일) 동안 **코드를 한 줄도 쓰지 않고**, 데이터 스키마(14테이블)와 이벤트 사전(13종)을 먼저 확정했다.

**해결하려는 문제**
이 연구의 최종 산출물은 웹앱이 아니라 **분석 가능한 연구 데이터**다. 그런데 개발은 보통 화면부터 시작한다.

**이 설계가 없으면**
화면을 먼저 만들고 나중에 데이터를 붙이면, **연구에 필요한 기록이 조용히 빠진다.** 예를 들어 "넛지가 언제 예정됐고 언제 실제로 노출됐는가"는 화면을 만들 땐 아무도 신경 쓰지 않는다. 실증이 끝나고 분석하려는 순간에야 "그 데이터가 없다"는 걸 알게 되고, **그때는 되돌릴 수 없다.** 데이터는 사후 소급 수집이 불가능하다.

**대안과 트레이드오프**
- 대안: 프로토타입을 빨리 만들고 데이터를 나중에 붙인다 → **속도는 빠르지만 연구 데이터 유실 위험**
- 채택: 데이터 먼저 → **초반이 느리고 "아무것도 안 만든 것처럼" 보인다**

**연구적 의미**
설계 순서를 **사용자 행동 흐름 → 저장할 이벤트 → 테이블·필드 → 코드**로 명문화했다([research-data.md](../.claude/rules/research-data.md)). 이는 연구용 시스템과 일반 앱을 가르는 첫 번째 분기점이다.

# 2. participant · session · message · event를 4층으로 분리했다

**결정**
네 층위를 명확히 구분해 기록한다. 혼용을 금지한다.

| 층위 | 의미 |
|------|------|
| **participant** | 연구 참여자 1명 (`P001`) |
| **session** | 한 번의 접속 (시작~종료) |
| **message** | 채팅창에 나타난 한 건의 메시지 |
| **event** | 클릭·화면열기·행동선택 등 시간순 사건 |

**해결하려는 문제**
일반적인 챗봇은 "대화 기록" 하나면 충분하다. 그러나 연구는 **서로 다른 분석 단위**를 요구한다.

**이 설계가 없으면**
- "참여자당 접속 횟수"를 셀 수 없다 (session이 없으면)
- "넛지를 보고 몇 초 뒤 반응했는가"를 알 수 없다 (event가 없으면)
- "출처를 클릭했는가"는 메시지가 아니라 **행동**이므로 message에 담을 수 없다

즉 **분석 단위가 뭉개지면 연구 질문 자체를 물을 수 없다.**

**대안과 트레이드오프**
- 대안: 모든 것을 `messages` 한 테이블에 JSON으로 → **저장은 간단하지만 집계·조인이 지옥**
- 채택: 4층 분리 → **테이블이 늘고 코드가 복잡해진다**

**연구적 의미**
횡단분석(참여자 간 비교)은 participant 단위, 종단분석(시간 변화)은 event 단위, 사용성 분석은 session 단위로 이루어진다. **분석 단위를 미리 데이터 구조에 새겨 넣은 것**이다.

# 3. 새로고침해도 세션이 중복 생성되지 않게 했다

**결정**
Streamlit은 상호작용마다 스크립트를 재실행(rerun)한다. 이 때문에 세션·메시지·이벤트가 중복 저장되지 않도록 **2겹으로 막았다.**
1. Streamlit 재실행 → `session_state`가 세션 ID를 들고 있어 재생성 안 함
2. 브라우저 새로고침(session_state 소실) → **DB에서 아직 안 끝난 세션을 찾아 이어받음**(2시간 이내)

**해결하려는 문제**
일반 개발자에게 이 문제는 "새로고침해도 오류 안 나면 OK"다. 연구자에게는 다르다.

**이 설계가 없으면**
참여자가 F5를 다섯 번 누르면 **접속 기록이 5건 늘어난다.** 그러면:
- "참여자별 접속 빈도"가 **부풀려진다**
- "세션 길이"가 **잘게 쪼개진다**
- 사용빈도와 자기효능감의 관계를 보려던 분석이 **애초에 성립하지 않는다**

**이건 버그가 아니라 측정 오류다.** 오류 메시지도 안 뜨고, 나중에 발견하기도 어렵다.

**대안과 트레이드오프**
- 대안: 새로고침 = 새 접속으로 간주 → **단순하지만 접속 횟수가 오염된다**
- 채택: 열린 세션 이어받기 → **"같은 접속"의 시간 창(현재 2시간)을 정해야 한다** (잠정값, 파일럿에서 조정)

**연구적 의미**
`sessions.started_at`/`ended_at`은 **사용시간·참여빈도라는 결과변수**의 원천이다. 여기가 오염되면 그 위에 쌓은 모든 분석이 무너진다.

# 4. `role`과 `message_type`을 분리했다

**결정**
메시지에 **누가 보냈나(`role`: user/assistant/system)** 와 **연구적으로 어떤 성격인가(`message_type`: nudge / nudge_response / rag_question / rag_answer / safety_message …)** 를 **별도 컬럼**으로 저장한다.

**해결하려는 문제**
챗봇에서는 "AI가 보낸 메시지"면 충분하다. 연구에서는 전혀 다르다.

**이 설계가 없으면**
AI가 보낸 메시지가 **넛지인지, RAG 답변인지, 안전 안내인지 구분할 수 없다.** 그러면:
- **넛지 반응률**을 계산할 수 없다 (어느 게 넛지인지 모르니까)
- **안전 응답 발생 빈도**를 셀 수 없다 → 과다 트리거 점검 불가
- "정보 중재 노출량"과 "행동 중재 노출량"을 분리할 수 없다

**대안과 트레이드오프**
- 대안: `role`만 저장하고 나중에 내용으로 추론 → **불가능하거나 부정확**
- 채택: 분리 저장 + DB CHECK 제약 → **타입을 늘릴 때마다 스키마 변경 필요**

**연구적 의미**
이 연구의 중재는 **두 종류**(선제적 넛지 / 근거 기반 답변)다. 둘의 효과를 구분해 보려면 데이터 층위에서 구분되어 있어야 한다.

# 5. 넛지의 '예정 시각'과 '실제 노출 시각'을 분리했다

**결정**
`nudge_events`에 `scheduled_at`(예정)과 `displayed_at`(실제 노출)을 **각각** 저장한다.

**해결하려는 문제**
넛지는 예정됐다고 반드시 노출되지 않는다. **사용자가 앱을 열지 않으면 노출되지 않는다.**

**이 설계가 없으면**
"중재를 몇 번 했는가"라는 질문에 답할 수 없다. **계획량(예정)과 실제 노출량이 뒤섞이기** 때문이다. 그러면:
- 넛지 반응률의 **분모가 틀린다** (예정 대비인가, 노출 대비인가?)
- "중재를 받았는데 반응 안 함"과 "애초에 중재를 못 받음"을 **구분할 수 없다**

**대안과 트레이드오프**
- 대안: 노출 시각만 저장 → 지금(접속시점 트리거)은 문제없지만, **예약 넛지로 확장하는 순간 붕괴**
- 채택: 분리 → **컬럼이 늘고, MVP에서는 두 값이 사실상 같다**(오버엔지니어링처럼 보임)

**연구적 의미**
"중재 노출과 행동 반응의 관계"가 이 연구의 핵심 질문이다. **노출을 정확히 세지 못하면 관계를 볼 수 없다.**

# 6. ⭐ 행동의도(하겠다)와 실제수행(했다)을 분리했다

**결정**
- `nudge_events.action_commitment` — 사용자가 **"하겠다"**고 선택한 행동
- `action_followups.completion_status` — 나중에 **"실제로 했다"**고 보고한 결과 (완료/일부완료/미완료/확인불가)
- 둘을 **연결 ID(`nudge_id`)** 로 잇는다

**해결하려는 문제**
행동과학에서 **의도(intention)와 수행(behavior)은 다른 구성개념**이다. 이 둘의 간극(intention-behavior gap)이 연구의 관심사다.

**이 설계가 없으면**
"넛지를 받고 하겠다고 했다"까지만 남고, **실제로 했는지는 영원히 모른다.** 그러면 이 연구는:
- **행동의도까지만** 측정하는 연구가 된다
- "넛지가 실제 행동 수행으로 이어지는가"라는 **가장 핵심적인 질문에 답할 수 없다**
- 의도→수행 **전환율**도, **소요시간**도 계산 불가

**대안과 트레이드오프**
- 대안: 한 필드에 "약속함/수행함"을 함께 → **두 시점의 사건을 하나로 뭉개 시간 차를 잃는다**
- 채택: 분리 + 연결 ID → **추후 확인 질문을 언제 던질지 결정해야 한다**(미결정 U7)

**연구적 의미**
이것이 **이 연구를 "챗봇 사용성 연구"가 아니라 "행동중재 연구"로 만드는 결정적 설계**다. 이 두 필드가 없으면 종단 행동분석 자체가 성립하지 않는다.

# 7. ⭐ 시스템 버전을 하나의 테이블로 묶어 모든 데이터에 연결했다

**결정**
`system_versions` 테이블 하나에 다음을 **묶어서** 관리하고, 모든 연구 기록(session · message · event · nudge · retrieval)이 이를 참조한다.

```
app_version · llm_model_version · embedding_version
prompt_bundle_version · nudge_rule_version
safety_rule_version · knowledge_base_version
activated_at · deactivated_at · change_note
```

**해결하려는 문제**
챗봇의 답변은 LLM 하나로 결정되지 않는다. **LLM + 임베딩 모델 + 프롬프트 + 지식베이스 + 안전 규칙 + 넛지 규칙**이 모두 결과에 영향을 준다. 그리고 연구 중간에 이 중 무엇이든 바뀔 수 있다.

**이 설계가 없으면**
연구 도중 프롬프트를 한 번 수정하는 순간, **참여자마다 서로 다른 중재를 받은 셈**이 된다. 그런데 그 사실이 데이터에 남지 않는다. 그러면:
- "P001은 좋았는데 P015는 나빴다"의 원인이 **참여자 차이인지 시스템 변경인지 알 수 없다**
- 논문에서 "버전 0.2에서 결과가 나빴다"고 할 때, **프롬프트가 바뀐 건지 지식베이스가 바뀐 건지 임베딩 모델이 바뀐 건지 특정할 수 없다**
- **연구 재현성이 무너진다** — 과거의 답변을 다시 만들어낼 수 없다

**대안과 트레이드오프**
- 대안: 코드 버전(git commit)만 기록 → **지식베이스·프롬프트 변경은 커밋에 안 잡힐 수 있다**
- 대안: 각 테이블에 개별 버전 컬럼 → **조합을 추적할 수 없다**
- 채택: 버전 조합을 하나의 ID로 → **설정을 바꿀 때마다 새 버전 행을 만드는 규율이 필요**

**연구적 의미**
이것은 **연구 재현성(reproducibility)과 내적 타당도(internal validity)를 위한 설계**다. 단일군 설계에서 시간에 따른 변화를 볼 때, "시스템이 바뀌지 않았음" 또는 "언제 어떻게 바뀌었음"을 **증명할 수 있어야** 결과를 해석할 수 있다. 일반 앱에서는 거의 하지 않는 설계이며, **연구용 시스템임을 가장 잘 보여주는 부분**이다.

# 8. ⭐ 검색 기록을 배열이 아니라 정규화했다 (연구팀 피드백)

**결정**
당초 설계는 검색 결과를 **배열**로 저장하려 했다.
```
retrieved_chunk_ids: [c1, c5, c9]
similarity_scores:   [0.91, 0.83, 0.77]
```
연구팀 리뷰를 받아 **검색 1회 = `retrieval_logs` 1행 / 검색된 청크 1개 = `retrieval_chunks` 1행**으로 바꿨다.

```
retrieval_chunks: retrieval_id · chunk_id · rank · similarity_score · was_selected
```

**해결하려는 문제**
"동작하면 된다"는 관점에서 배열은 문제없다. 답변을 만드는 데는 충분하다. **문제는 분석이다.**

**이 설계가 없으면 (배열로 저장하면)**
아래 질문들에 **SQL로 답할 수 없다** (또는 극도로 어렵다):
- **어떤 문서 청크가 몇 번 검색됐는가?** → 지식베이스에서 실제로 쓰이는 자료 vs 죽은 자료 파악 불가
- **유사도 점수 분포는?** → 근거 부족 임계값을 어디에 둘지 결정할 근거 없음
- **검색된 청크 중 실제로 답변에 쓰인 비율은?** → 검색 품질과 생성 품질을 **분리 평가할 수 없음**
- **순위별 채택률은?** → top-k를 몇으로 둘지 판단 불가

즉 우리 자신의 규칙([RAG_RULES.md](../RAG_RULES.md))인 **"검색 품질과 답변 품질을 분리 평가한다"가 실행 불가능**해진다.

**대안과 트레이드오프**
- 대안(원안): 배열 → **저장이 간단하고 코드가 짧다. 그러나 분석이 사실상 불가능**
- 채택: 정규화 → **행 수가 늘고(질문 1건당 top-k개) 조인이 필요**

**연구적 의미**
RAG는 할루시네이션을 **줄이는** 장치일 뿐 제거하지 못한다. 오답이 나왔을 때 **검색 실패인지 생성 실패인지 구분**하려면 검색 과정 자체가 데이터로 남아야 한다. 이 정규화가 그것을 가능하게 한다.

> **이 항목은 연구팀 리뷰가 잡아낸 것이다.** 원 설계자(AI)는 "동작하는" 구조를 만들었고, 연구자가 "분석 가능한" 구조를 요구했다. 이 차이가 연구용 시스템의 본질이다.

# 9. 기술 오류를 연구 행동 데이터와 분리했다

**결정**
API 실패·DB 실패·검색 실패를 **`technical_errors` 테이블에 별도** 기록한다. 연구 행동 데이터(events/messages)와 섞지 않는다.

**해결하려는 문제**
참여자가 넛지에 응답하지 않았다. **왜인가?**

**이 설계가 없으면**
"무응답"이라는 하나의 관측치만 남는다. 그런데 그 무응답은 두 가지 완전히 다른 의미일 수 있다.
- **실제 무관심** → 연구적으로 의미 있는 행동 데이터
- **시스템 실패** (앱이 오류로 넛지를 못 띄웠음) → **연구 데이터가 아니라 결측(missing)**

이 둘을 구분하지 못하면 **넛지 반응률이 시스템 안정성에 오염된다.** 시스템이 자주 죽으면 "참여자가 넛지에 관심 없다"는 결론이 나온다. **명백한 오귀인(misattribution)이다.**

**대안과 트레이드오프**
- 대안: 오류를 로그 파일로만 → **연구 데이터와 시간축으로 대조할 수 없다**
- 채택: DB에 참여자·세션과 연결해 저장 → 테이블이 하나 늘어난다

**실제 사례 (2026-07-13)**
앱에서 "대화를 불러오지 못했습니다" 오류가 발생했다. `technical_errors`에 남은 로그로 **원인을 정확히 특정**했다 — 코드 버그가 아니라 서버가 옛 모듈을 붙들고 있었던 것. **이 테이블이 없었으면 추측으로 코드를 뒤졌을 것이다.** 승인받아 만든 지 30분 만에 값을 했다.

**연구적 의미**
결측 데이터의 **메커니즘을 식별**할 수 있게 한다. 이는 분석 시 결측 처리 방식(MCAR/MAR/MNAR 판단)에 직접 영향을 준다.

# 10. 연구 데이터를 외부에서 쓸 수 없게 DB 차원에서 막았다

**결정**
- 전 테이블 **RLS 활성화 + 정책 0개** → 외부(anon/authenticated)는 읽기·쓰기 **전면 차단**
- 앱 서버만 secret 키로 우회 (Streamlit은 100% 서버 실행 → 키가 브라우저로 나가지 않음)
- **복합 외래키**로 "P001 세션인데 이벤트는 P002" 같은 오염을 **DB가 물리적으로 거부**
- `role`·`status` 등에 **CHECK 제약** → 오타가 데이터를 오염시키지 못함

**해결하려는 문제**
연구 데이터의 **무결성**은 앱 코드의 규율에 맡길 수 없다.

**이 설계가 없으면**
- 앱 주소를 아는 사람이 **가짜 참여자 데이터를 밀어넣을 수** 있다 → 연구 데이터 오염
- 코드 실수 한 번으로 참여자·세션이 어긋난 행이 저장된다 → **조용히 오염되고 나중에 발견하기 어렵다**

**대안과 트레이드오프**
- 대안: 앱 코드에서 검증 → **코드를 고칠 때마다 다시 실수할 수 있다**
- 채택: DB가 강제 → **제약을 바꾸려면 마이그레이션이 필요하다**(의도된 마찰)

**연구적 의미**
"데이터가 오염되지 않았다"를 **주장이 아니라 구조로 보증**한다. IRB·감사 대응에서도 근거가 된다.

# 11. 넛지를 LLM 자유생성이 아니라 승인 템플릿으로 만들었다

**결정**
연구자가 승인한 **템플릿 v0.1 (5종)** 을 시간대 규칙으로 선택해 **그대로** 제시한다. MVP에서는 LLM 문장화를 쓰지 않는다.

**해결하려는 문제**
LLM에게 넛지를 생성시키면 **참여자마다 다른 문장**을 받는다.

**이 설계가 없으면**
- 참여자 A는 따뜻한 문장을, 참여자 B는 딱딱한 문장을 받는다 → **중재가 표준화되지 않는다**
- "넛지가 효과 있었다"고 해도 **어떤 넛지가 효과 있었는지 말할 수 없다**
- 단일군 설계에서 **중재 자체가 변수가 되어버린다** → 결과 해석 불가

**대안과 트레이드오프**
- 대안: LLM 자유생성 → **자연스럽지만 중재 표준화 실패 + 비용 발생 + 안전 문구 통제 불가**
- 채택: 템플릿 → **문장이 반복돼 지루할 수 있다** (파일럿에서 확인 필요)

**부수 효과**
넛지에 LLM을 호출하지 않으므로 **AI 비용이 0원**이다. 전체 API 비용을 크게 낮췄다.

**안전 측면**
템플릿이므로 **죄책감·공포·강압 표현을 사전에 배제**할 수 있다. 특히 야식 넛지는 "야식 드시면 안 돼요"(훈계) 대신 **"저녁 드신 뒤에 뭔가 더 드셨어요?"**(사실 확인)로 작성했다. LLM 자유생성이었다면 이 통제가 불가능하다.

**연구적 의미**
`template_key`·`template_version`을 함께 저장하므로, **템플릿별 반응률 비교**가 가능하다. 즉 "어떤 넛지가 효과적인가"를 물을 수 있다.

# 12. 스스로에게 500줄 제한을 걸었다

**결정**
핵심 실행 Python 코드(app·rag·nudge·database·safety·config)를 **500줄 이내**로 유지한다. (빈 줄·주석 제외 기준 — 2026-07-13 확정)

**해결하려는 문제**
비개발자 연구자가 **바이브코딩**으로 개발한다. 코드를 이해하지 못하면 통제할 수 없다.

**이 설계가 없으면**
AI가 코드를 계속 부풀린다. 어느 순간 연구자가 **자기 시스템을 이해하지 못하게 된다.** 그러면:
- 무엇이 잘못됐는지 판단할 수 없다
- AI가 "됐습니다"라고 하면 믿는 수밖에 없다
- **연구자가 자기 연구도구를 통제하지 못한다**

**대안과 트레이드오프**
- 대안: 제한 없음 → **기능은 빨리 늘지만 이해 가능성을 잃는다**
- 채택: 500줄 → **기능을 넣을 때마다 "이게 정말 필요한가"를 묻게 된다**(의도된 마찰)

**기준을 '주석 제외'로 정한 이유**
전체 574줄 vs 실제 코드 332줄로 갈렸다. 규칙의 **목적은 시스템 단순화이지 설명 축소가 아니므로**, 비개발자가 코드를 읽을 수 있도록 **한국어 주석은 충분히 남기고** 실제 코드만 센다. 측정 도구(`tests/check_line_limit.py`)를 만들어 자동 확인한다.

**연구적 의미**
연구도구의 **투명성과 통제 가능성**. 논문에서 시스템을 기술할 때, 500줄짜리 시스템은 부록에 전체를 실을 수도 있다.

# 부록. 이 설계가 만든 것 — 한 문장씩

| 설계 | 없으면 잃는 것 |
|------|---------------|
| 데이터 먼저 | 사후 소급 불가능한 연구 데이터 |
| 4층 분리 | 분석 단위 (횡단/종단/사용성) |
| 세션 중복 방지 | 접속 빈도·사용시간의 정확성 |
| role × message_type | 넛지 반응률, 안전 응답 빈도 |
| 예정 ≠ 노출 | 중재 노출량의 정확한 분모 |
| **의도 ≠ 수행** | **행동중재 연구 자체** |
| **system_versions** | **연구 재현성·내적 타당도** |
| **검색 정규화** | **검색 실패 vs 생성 실패 구분** |
| technical_errors | 결측 메커니즘 식별 |
| RLS + 제약 | 데이터 무결성 보증 |
| 템플릿 넛지 | 중재 표준화 |
| 500줄 제한 | 연구자의 시스템 통제력 |

이 문서의 모든 주장은 **아직 가설이다.** 설계가 정말 옳았는지는 다음으로만 증명된다.

> **실증이 끝난 뒤, 이 데이터로 [analysis-targets.md](analysis-targets.md)에 정의한 결과물을 실제로 만들어낼 수 있는가?**

만들어낼 수 있다면 이 설계는 정당했다. 만들어내지 못한다면, 아무리 정교해도 실패한 설계다.
**설계의 가치는 우아함이 아니라 연구 질문에 답하는 능력으로 판정된다.**

# NUDGE_RULES.md — 넛지 규칙

> **문서 버전:** v0.1 · **작성 기준일:** 2026-07-09
> 이 문서는 넛지를 **언제·어떻게 제시하고, 어떻게 반복을 제한하는지**를 정의한다.
> 넛지는 [.claude/rules/medical-safety.md](.claude/rules/medical-safety.md)의 안전 규칙을 넘지 않는다. 데이터는 [DATA_SCHEMA.md](DATA_SCHEMA.md)의 `nudge_events`·`action_followups`, 이벤트는 [EVENT_DICTIONARY.md](EVENT_DICTIONARY.md)를 따른다.
> 수치 기준(하루 허용 횟수·간격·임계값 등)은 아직 **미결정**이며, 연구팀이 확정하기 전까지 아래 값은 **잠정 기본값**이다.

---

## 1. 넛지 행동 영역

넛지가 다루는 건강행동 영역은 넷이다.

| 영역 | 예시 |
|------|------|
| **식사** | "점심 식사는 하셨어요?" |
| **운동** | "식사 후 10분 정도 가볍게 걸어보시는 건 어떠세요?" |
| **복약** | "혹시 오늘 약은 챙겨 드셨어요?" (복약 상기까지만 — 용량·종류 판단 금지) |
| **혈당관리** | "오늘 식전 혈당은 어떠셨나요?" |

> **MVP 시작 영역: 식사** (2026-07-13 연구팀 확정). 간식·야식을 포함해 **혈당 변동에 직결되는 섭식 행동**을 먼저 다룬다. 운동·복약·혈당측정은 이후 확장.
> **복약 넛지는 "챙기셨어요?" 수준의 행동 상기까지만.** 용량·종류·중단 등 의료적 판단으로 확장하지 않는다(안전 규칙).

## 1-1. 승인 템플릿 v0.1 (2026-07-13 연구팀 승인 — 임의 수정 금지)

문구를 바꾸려면 **`TEMPLATE_VERSION`을 올리고** 이 표를 갱신한다. 모든 참여자가 동일한 자극을 받아야 결과 해석이 가능하다.

| 시간대 | template_key | 문구 | 응답 선택지 |
|--------|--------------|------|-------------|
| 06–10시 | `meal_breakfast` | 안녕하세요. 오늘 아침 식사는 하셨어요? | 네, 먹었어요 / 아직이요 / 나중에 답할게요 |
| 11–14시 | `meal_lunch` | 점심 식사는 하셨어요? | 네, 먹었어요 / 아직이요 / 나중에 답할게요 |
| 14–17시 | `snack_afternoon` | 오후에 간식 드셨어요? | 네, 먹었어요 / 아니요 / 나중에 답할게요 |
| 17–20시 | `meal_dinner` | 저녁 식사는 하셨어요? | 네, 먹었어요 / 아직이요 / 나중에 답할게요 |
| **21시 이후** | `snack_late` | 저녁 드신 뒤에 뭔가 더 드셨어요? | 네, 먹었어요 / 아니요 / 나중에 답할게요 |

- **넛지 없는 시간대:** 00–06시, 10–11시, 20–21시 → 억지로 띄우지 않는다.
- **야식 넛지의 문구 원칙:** "야식 드시면 안 돼요" 같은 **훈계·죄책감 유발 표현을 금지**한다. 판단하지 않고 **사실만 묻는다.** 넛지의 목적은 혼내는 것이 아니라 스스로 인식하도록 돕는 것이다.
- MVP는 **LLM 문장화 없이 템플릿 그대로** 사용한다(일관성·비용).

## 2. 초기 넛지 트리거 (MVP: 웹앱 접속 시점)

MVP는 예약·푸시 없이 **사용자가 웹앱을 열 때** 넛지를 판단한다. 판단 시 확인하는 것:

1. **웹앱 접속 시각 / 현재 시간대** (예: 12:10 접속 → 점심 관련)
2. **당일 넛지 노출 여부** (오늘 이미 같은 넛지를 봤는지)
3. **직전 사용자 응답** (방금 거절/완료했는지)
4. **미완료 행동 약속** (이전에 "하겠다"고 한 것)
5. **후속 확인 필요 여부** (약속한 행동의 수행을 물어볼 때가 됐는지)

```
사용자 접속
 → 현재 시간·시간대 확인
 → 최근 넛지·행동 기록 확인 (반복 제한 규칙 적용)
 → 해당 시간대 넛지 규칙 선택
 → 승인된 템플릿 표시 (필요 시에만 LLM 문장화)
```

## 3. 넛지 선택 방식 — 규칙·템플릿 우선

- 연구자가 **승인한 넛지 유형과 문장 템플릿**을 우선 사용한다.
- 시간·행동·직전 응답에 따라 **규칙으로 템플릿을 선택**한다.
- **필요할 때만** LLM이 자연스러운 문장으로 다듬는다. (모든 넛지에 LLM을 호출하지 않는다 — 비용·일관성)
- 참여자마다 전혀 다른 중재를 받으면 결과 해석이 어려우므로 **자유 생성을 제한**한다.
- `nudge_type` 예: 정보형 / 선택구조형 / 작은행동제안 / 자기효능감강화.

## 4. 반복 제한 규칙 (잠정 기본값 — 연구팀 확정 필요)

과도한 넛지는 피로·거부감을 부른다. 다음으로 제한한다.

| 규칙 | 값 |
|------|------------|
| 동일 넛지의 하루 최대 노출 횟수 | **1회** ✅ 구현됨 |
| 응답 완료 후 당일 재노출 | **금지** ✅ 구현됨 |
| 거절·"나중에" 직후 같은 넛지 재권유 | **금지** (당일) ✅ 구현됨 — 사용자의 거절 의사를 존중한다 |
| **하루 전체 넛지 최대 노출** | **3회** ✅ (2026-07-13 연구팀 확정) |
| 넛지 간 최소 시간 간격 | 시간대가 겹치지 않으므로 별도 간격 규칙 불필요 |
| 새로고침(rerun)으로 인한 중복 노출 | **방지** ✅ 구현됨 (기술적 필수) |

> **하루 3회 상한을 둔 이유:** 시간대가 5개라 이론상 최대 5회지만, 과도한 넛지는 피로·거부감을 부른다. 넛지는 **앱을 열 때만** 뜨므로 실제 노출은 더 적다.

- **새로고침 중복 방지**는 값이 아니라 구현 필수사항이다. 같은 예정 넛지가 rerun으로 두 번 노출·기록되지 않도록 `nudge_events`의 `displayed_at`·`status`로 멱등 처리한다.
- 수치는 실증 전 파일럿에서 조정할 수 있게 `config`/`system_versions`로 관리한다.

## 5. 메시지 작성 원칙

- **한 번에 하나의 질문·하나의 행동**만.
- **짧고 쉬운 문장** (고령친화적).
- **명령보다 제안형** 표현. ("걸으세요" ✕ → "가볍게 걸어보시는 건 어떠세요?" ○)
- **죄책감·공포·강압 표현 금지.** ("안 하면 위험해요" ✕)
- **거절 또는 나중에 응답할 선택지**를 항상 제공. ("아니요 / 나중에" 버튼)
- **승인된 템플릿을 우선** 사용. (문구 변경은 템플릿 버전으로 관리)

## 6. 행동 약속과 추후 확인 (종단)

- 사용자가 "하겠다"고 하면(`action_committed`) `action_followups`에 예정 시각과 함께 저장한다.
- 다음 적절한 접속 시 수행 여부를 묻고(`action_completed`), **완료 / 일부완료 / 미완료 / 확인불가**로 기록한다.
- **의도(하겠다)와 실제 수행(했다)을 분리**해 종단 분석에 쓴다.
- 재확인 시점(며칠 뒤, 다음 접속 시 등)은 **미결정** — 연구팀 확정.

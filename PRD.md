# PRD — 제품 · 연구 요구사항

> **문서 버전:** v0.1 (초안) · **작성 기준일:** 2026-07-07 · **상태:** Phase 0 작성 중
> 이 문서는 "무엇을 만들고 무엇을 만들지 않을지"를 정의한다. 개발 중 범위 변경은 이 문서를 먼저 갱신한 뒤 진행한다.
>
> **⚠️ 방향 전환 (2026-07-19 · DECISIONS D44/D45):** 연구팀 논의로 **페이퍼의 주 데이터원 = 사용 후 설문(사용 경험)** 으로 정해졌다. 이에 따라 우선순위가 **데이터 정밀도 < 완성도·친숙도·사용성** 으로 이동하고, 대화 히스토리를 **지속 노출**(2시간 리셋 폐기, ChatGPT식 새 대화+사이드바)한다. 상호작용 로그는 **최대한 확보하되 부차**. 아래 본문의 "과정 데이터가 최종 산출물" 서술은 이 전환에 맞춰 §12·부록 B에서 보정한다.

## 1. 프로젝트 배경

고령 제2형 당뇨 환자는 식사 · 운동 · 복약 · 혈당관리 같은 일상적 자기관리 행동을 꾸준히 유지하기 어렵다.
정보가 부족해서라기보다, 적절한 시점에 작은 행동을 떠올리고 실행하도록 돕는 계기가 부족한 경우가 많다.
또한 인터넷의 건강정보는 출처가 불분명하거나 개인에게 부적절할 수 있어 신뢰하기 어렵다.

이 프로젝트는 **선제적 넛지**(적절한 시점의 행동 유도)와 **근거 기반 RAG 채팅**(신뢰 문서에 기반한 답변)을
하나의 대화 화면에서 제공하고, 이것이 사용자의 행동과 태도에 갖는 관계를 실증적으로 관찰한다.

## 2. 해결하려는 문제

1. 고령 당뇨 환자가 자기관리 행동을 **적절한 시점에 떠올리고 실행**하도록 돕는 계기가 부족하다.
2. 당뇨 관련 건강정보의 **출처와 신뢰성**을 사용자가 확인하기 어렵다.
3. 넛지 · 정보 제공이 실제 사용자 반응(이해 · 신뢰 · 자기효능감 · 행동)과 어떤 관계를 갖는지에 대한
   **현장 실증 데이터**가 부족하다.

## 3. 주요 사용자 정의

| 구분 | 정의 | 특성 |
|------|------|------|
| **참여자 (주 사용자)** | 고령 제2형 당뇨 환자 | 스마트폰/태블릿에 익숙하지 않을 수 있음. 작은 글씨 · 복잡한 화면에 취약. 익명 `participant_id`로 관리. |
| **연구자** | 연구를 설계 · 운영하는 사람 | 참여자 등록, 넛지 · 문서 승인, 데이터 확인 및 분석 수행. |

## 4. 연구용 웹앱의 목적

- AI 모델 개발 · fine-tuning · 멀티에이전트 검증이 **아니다**.
- 근거 기반 정보 제공과 행동 넛지가 사용자의 **이해 · 신뢰 · 자기효능감 · 행동의도 · 행동 수행**에
  갖는 관계를 살펴보는 **사회 · 행동과학 실증**이다.
- 실험군 · 대조군 없이 단일군의 **횡단 분석**(시점 간 참여자 비교)과 **종단 분석**(같은 사람의 시간 변화)을 함께 고려한다.
- 대조군이 없으므로 강한 인과 표현("시스템이 행동을 바꿨다")보다, **중재 노출과 행동 반응의 관계 · 사용 전후 변화 · 현장 수용성**을 중심으로 해석한다.

## 5. 핵심 사용자 시나리오

```
참여자가 웹앱 접속 (참여자 코드 입력)
  → 세션 시작
  → 현재 시간대에 맞는 넛지 표시   (예: 정오 → "점심 식사는 하셨어요?")
  → 사용자가 넛지에 응답 (반응 저장)
  → 사용자가 당뇨 질문 입력          (예: "식후 혈당은 언제 재는 게 좋아요?")
  → 승인 문서에서 근거 검색
  → 근거 기반 답변 + 출처 표시
  → 질문 · 검색근거 · 답변 · API 비용 저장
  → (행동 약속이 있으면) 다음 접속 시 수행 여부 확인
  → 연구자가 데이터를 CSV로 확인
```

넛지 화면과 질문 화면을 오갈 필요 없이 **하나의 대화 흐름** 안에서 진행된다.

## 6. 선제적 넛지 기능 정의

- **목적:** 정보 전달이 아니라, 사용자가 작은 행동을 **떠올리고 · 선택하고 · 실행**하도록 돕는다.
- **행동 영역:** 식사 · 운동 · 복약 · 혈당관리 _(MVP 시작 영역은 미결정 — 식사 · 운동 우선 검토)_
- **MVP 트리거:** 웹앱 접속 시점. 접속 시각 · 최근 기록 · 시간대 규칙을 확인해 승인 템플릿을 표시한다.
- **생성 방식:** 연구자가 승인한 넛지 유형과 문장 템플릿을 우선 사용하고, 필요할 때만 LLM이 자연스러운 문장으로 조정한다.
  (참여자마다 전혀 다른 중재를 받으면 결과 해석이 어려워지므로 자유 생성을 제한한다.)
- **반복 제한:** 동일 넛지의 하루 최대 노출, 응답 후 재노출 제한, 거절 직후 재권유 제한, 새로고침 중복 방지 등은 `NUDGE_RULES.md`에서 정한다.

## 7. RAG 질문 · 답변 기능 정의

- **처리 흐름:** 질문 입력 → 위험도 사전 확인 → 질문 임베딩 → 문서 청크 검색 → 근거 충분성 확인 → 근거 기반 답변 생성 → 출처 표시 → 전 과정 저장.
- **문서 범위:** 연구자가 승인한 신뢰 문서만 사용한다. 문서명 · 발행기관 · 발행연도 · 버전을 기록하며, 개정 시 삭제하지 않고 새 버전으로 관리한다.
- **답변 규칙:** 검색된 근거를 우선 사용하고, 근거에 없는 내용은 추측하지 않는다. 근거가 부족하면 답변을 보류하고 안내한다.
- **한계 인식:** RAG는 할루시네이션을 **줄이는** 장치이지 완전히 제거하는 장치가 아니다. 승인 문서 · 근거 부족 시 보류 · 출처 저장 · 안전 질문 분리 · 테스트 질문 검증을 함께 둔다.

## 8. MVP 포함 기능

첫 버전에서 **하나의 완전한 사용자 흐름**을 끝까지 구현한다.

- [ ] 참여자 코드 입력 · 참여자 확인 · 세션 생성
- [ ] 하나의 채팅 화면
- [ ] 접속 시점 기반 시간대 넛지 (초기 1종 → 식사 · 운동 확장)
- [ ] 넛지 응답 · 행동 약속 저장
- [ ] 다음 접속 시 행동 수행 여부 확인 (종단 추적)
- [ ] 사용자 자유 질문 (RAG)
- [ ] 승인 문서 검색 · 근거 기반 답변 · 출처 표시
- [ ] 위험 질문에 대한 제한 응답 (안전성 처리)
- [ ] 질문 · 검색근거 · 답변 · API 토큰 · 비용 저장
- [ ] 참여자별 이벤트의 시간순 저장
- [ ] 연구 데이터 CSV 추출
- [ ] **연구자용 관리자 대시보드 (최소판)** — 아래 §8-1

## 8-1. 연구자용 관리자 대시보드 (2026-07-13 범위 추가)

**왜 필요한가 (연구 운영상 실제 필요):**
1. **공동연구자는 비개발자다.** Supabase 대시보드로 원시 테이블을 보게 할 수 없다. 연구 지표를 **바로 읽을 수 있는 화면**이 필요하다.
2. **학교·기관과 결과를 공유**해야 한다. 시각화된 요약이 있어야 보고·발표가 가능하다.

**보여줄 것 (근거: [docs/analysis-targets.md](docs/analysis-targets.md)의 "최소 결과물")**
- 참여자별 **접속 횟수 · 세션 길이**, 참여 현황
- **넛지 노출 → 반응 전환율** (템플릿별 비교 포함)
- ⭐ **행동의도 → 실제 수행 전환율 · 소요시간**
- **질문 수 · 출처 클릭률**
- **안전 응답 발생 빈도·유형** (과다 트리거 점검)
- (설문 도입 시) 시점별 측정값 변화
- 운영: **API 누적 비용**, 기술 오류 발생
- **CSV 추출** (기존 MVP 항목 통합)

**설계 원칙 (중요)**
- **참여자용 앱과 완전히 분리**한다. 별도 주소 · 연구자만 접근. 절대 같은 URL로 노출하지 않는다.
- **연구 행동 데이터(messages · events · nudge_events · retrieval_logs)는 읽기 전용.** 수정·삭제 기능을 두지 않는다(데이터 무결성).
  - 예외: **참여자 등록 · 상태 변경**(scheduled/active/completed/withdrawn)만 허용.
- 참여자는 **익명 ID로만** 표시한다(직접식별정보 없음).
- 관리자 도구는 **핵심 실행 코드 500줄 제한의 대상이 아니다.** 참여자용 앱과 성격이 다르므로 별도 앱으로 분리한다. → [DECISIONS.md](DECISIONS.md) D22

**범위 구분 (일정 현실성)**
- **MVP(8월 말):** 위 지표의 **최소판** — 핵심 숫자 + 간단한 표/그래프 + CSV 추출
- **MVP 이후(Phase 5):** 학교 공유용 리포트 자동 생성, 고급 시각화, 실시간 이상 징후 알림

## 10. 고령친화적 UI 원칙

- 한 화면에 한 번에 **하나의 질문 · 하나의 행동**만 제시한다.
- 짧고 쉬운 문장을 사용한다. 명령보다 **제안형** 표현을 쓴다.
- 큰 글씨 · 충분한 여백 · 큰 터치 영역을 유지한다.
- 답변은 고령친화적으로 **짧게** 유지한다.
- 스마트폰 · 태블릿 · 노트북에서 같은 주소로 접속 가능한 반응형 화면.
- 거절하거나 나중에 응답할 선택지를 항상 제공한다. 죄책감 · 공포 · 강압 표현을 쓰지 않는다.

## 11. 의료적 안전성 원칙

- 개인별 **진단 · 처방 · 복약(용량) 변경**을 직접 지시하지 않는다.
- 근거가 부족하면 추측하지 않고 답변을 보류한다.
- 응급 · 고위험 상황이 감지되면 일반 RAG 흐름보다 **고정된 안전 안내**를 우선한다. (안내 문구는 의료전문가 검토 필요)
- 의료진 · 약사 확인이 필요한 질문에는 상담을 권고한다.
- 안전 응답이 작동하면 `safety_message_shown` 이벤트와 질문 범주를 기록해 사후 검토한다.
- 세부 규칙은 `SAFETY_RULES.md`에서 정한다.

## 12. 연구 데이터 수집 범위

> **방향 전환 반영 (2026-07-19 · D44):** 페이퍼의 주 데이터원은 **사용 후 설문(사용 경험)** 이다. 아래 과정 데이터(로그)는 **계속·최대한 수집하되** 주 분석 대상이 아니라 보조·예비다("확보 후 사용 여부 결정"). 따라서 설문/측정(`measurements`) 수집 흐름이 **최우선 구현 대상**이며, 로그 완결성을 위해 기능을 늦추지 않는다.

**최종 설문 점수만으로는 부족하다.** 행동 변화의 흐름을 분석하려면 과정 데이터가 함께 있어야 한다.

- 참여자 · 세션 · 메시지 · 이벤트
- 넛지 예정/노출/반응, 행동 약속과 실제 수행 확인
- RAG 검색 근거(retrieval), 문서 · 청크
- 설문 · 반복 측정값
- 외부 API 호출 · 토큰 · 비용, 시스템 버전, 기술 오류

원칙:
- 모든 사건에 **정확한 시각 · 참여자 ID · 시스템 버전**을 기록한다.
- **행동의도(하겠다)와 실제 수행(했다)을 분리**해 기록한다.
- 실명 · 전화번호 · 주소 · 환자번호 등 직접식별정보는 저장 · LLM 전송하지 않는다. 신원 연결표는 애플리케이션 데이터와 분리한다.
- 영구 연구 데이터는 Supabase에 저장한다. (화면 임시 상태와 분리)
- 세부 정의는 `DATA_SCHEMA.md`, `EVENT_DICTIONARY.md`에서 정한다.

## 13. 8월 말 완성 기준 (MVP 완료 기준)

- [ ] 참여자 1명이 처음부터 끝까지 사용할 수 있다.
- [ ] 넛지와 자유 질문이 한 채팅 화면에서 작동한다.
- [ ] RAG 답변에 사용된 문서 청크가 기록된다.
- [ ] 근거 부족 시 제한 응답이 작동한다.
- [ ] 참여자별 이벤트가 시간순으로 저장된다.
- [ ] API 토큰과 예상 비용이 저장된다.
- [ ] 연구자가 데이터 파일을 내보내어 분석 구조를 확인할 수 있다.

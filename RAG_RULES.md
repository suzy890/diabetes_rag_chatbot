# RAG_RULES.md — 검색·답변 규칙

> **문서 버전:** v0.1 · **작성 기준일:** 2026-07-09
> 이 문서는 RAG(근거 기반 답변)가 **어떤 문서를 쓰고, 어떻게 검색·판단·답변하는지**를 정의한다.
> 안전 판단은 [SAFETY_RULES.md](SAFETY_RULES.md)를 우선한다. 검색 과정은 [DATA_SCHEMA.md](DATA_SCHEMA.md)의 `retrieval_logs`에 저장한다.
> 검색 임계값·top-k 등 수치는 **미결정**이며 아래는 잠정 기본값이다.

---

## 1. 문서 범위 (지식베이스)

- **연구진이 승인한 문서만** 운영 지식베이스에 포함한다.
- 각 문서는 **문서명·발행기관·발행연도·버전**을 기록한다(`documents`).
- **비승인 문서는 제외**한다.
- 문서가 개정되면 **삭제하지 않고 새 버전으로 등록**하고, 이전 버전은 비활성화(`is_active=false`)한다. → 과거 답변 재현 가능.

## 2. RAG 처리 흐름

```
사용자 질문
 → ① 위험도 확인 (SAFETY_RULES 우선 — 고위험이면 여기서 안전 처리로 분기)
 → ② 질문 임베딩 (model_calls 기록)
 → ③ 문서 청크 검색 (top-k)
 → ④ 근거 충분성 확인 (아래 3항 임계값)
 → ⑤ 근거 기반 답변 생성 (근거 범위 안에서만)
 → ⑥ 출처 표시
 → ⑦ 전 과정 저장 (retrieval_logs: 검색 청크·점수·선택 청크·top_k·KB버전)
```

## 3. 근거 충분성 판단 — 3단계 (과도한 보류 방지)

> **연구팀 피드백 반영:** 답변 보류가 실증 때 **과하게 자주 뜨면** 사용자 경험·신뢰가 떨어진다. 안전은 유지하되, "근거 없음 → 무조건 거절"로 남발되지 않도록 **3단계**로 나눈다.

| 단계 | 조건(잠정) | 처리 |
|------|-----------|------|
| **충분** | 최상위 청크 유사도 ≥ 상한 임계값 | 근거 기반으로 정상 답변 + 출처 |
| **부분** | 하한 ≤ 유사도 < 상한 | **완전 거절 대신** 찾은 범위의 일반 정보를 제공하되 **"이 부분은 일반적 정보이며 자세한 건 의료진과 상의하세요"** 라고 한계를 명시 |
| **부족** | 유사도 < 하한 임계값 | 추측하지 않고 "현재 지식베이스에서 충분한 근거를 찾지 못했다"고 안내(보류) |

- **상·하한 임계값과 top-k는 미결정** — 파일럿에서 튜닝. `config`/`system_versions`로 관리.
- 과도한 보류를 줄이는 실질 방안:
  - **지식베이스 커버리지를 충분히 확보**한다(자주 나올 질문을 미리 문서로 커버).
  - 임계값을 **너무 높게 잡지 않는다**(너무 높으면 보류 남발, 너무 낮으면 부정확 — 균형).
  - **보류율을 모니터링**한다(`retrieval_logs`로 보류 빈도·주제를 집계해 문서·임계값 개선).

## 3-1. 용어 모호성 처리 (2026-07-14 추가)

> **배경:** 고령 참여자는 임상 용어를 정확히 쓰지 않는다. "메트포르민"을 그냥 **"약"**이라 하고, **"혈당"**이라는 한 단어로 공복혈당·식후혈당·**당화혈색소**를 모두 가리킬 수 있다.
> 그런데 이 셋은 **의학적으로 완전히 다른 지표**다. (혈당 = 그 순간의 값 / 당화혈색소 = 최근 2~3개월 평균 상태를 반영)

### 왜 이것이 검색 문제가 아니라 **안전 문제**인가

> 참여자: **"혈당이 7인데 괜찮은가요?"**
> — **당화혈색소 7%** 로 읽으면 "관리 목표 근처"
> — **공복혈당 7.0 mmol/L**(=126 mg/dL) 로 읽으면 전혀 다른 이야기

**같은 숫자가 정반대로 해석된다.** 시스템이 **추측해서 하나를 고르면**, 관리가 안 되는 참여자에게 "괜찮다"고 안심시킬 수 있다.
→ **모호할 때 추론하지 않는다. 되묻는다.** (근거 부족 시 추측 금지 원칙의 연장)

### 두 가지 문제를 구분한다

| 유형 | 내용 | 처리 |
|------|------|------|
| **A. 어휘 격차** | 사용자 표현 ≠ 문서 용어 ("약" ↔ "메트포르민") | **승인 용어 사전**으로 매핑 + **하이브리드 검색**(벡터+키워드) |
| **B. 의미 모호성** | 한 단어가 서로 다른 개념을 지칭 ("혈당" ↔ 공복/식후/당화혈색소) | **되묻기(clarification)** — 추측 금지 |

### 승인 용어 사전

- 연구자가 **승인한 매핑표**를 사용한다. LLM이 임의로 판단하지 않는다. → [prompts/term-aliases.md](prompts/term-aliases.md)
- **버전 관리한다.** `system_versions`에 연결해 "어떤 용어사전으로 중재했는가"를 추적한다(재현성).
- ⚠️ **사전에는 임상 수치·기준을 넣지 않는다.** 용어가 **무엇을 가리키는 말인지**만 매핑한다. 수치·목표값은 **승인 문서에서 검색해서** 답한다. (근거 기반 원칙 유지)
- ⚠️ **의료전문가 검토 필수.**

### 되묻기(clarification) 규칙 — 남발 금지

**연구팀 피드백 반영:** 안전장치가 과하게 뜨면 사용자 경험·신뢰가 무너진다. 되묻기도 마찬가지다.

| 되묻는다 ✅ | 되묻지 않는다 ❌ |
|------------|-----------------|
| **모호성이 답을 실제로 바꾸는 경우** (예: "혈당이 7인데 괜찮나요?") | 일반 질문 (예: "혈당 관리는 어떻게 하나요?") |
| 수치·판단이 걸린 질문 | 모호해도 답이 같은 경우 |
| 안전 관련 질문 | 이미 되물어 답을 받은 경우(같은 세션 내) |

**되묻기 형식** (고령친화 — 한 번에 하나, 선택지 제공, "모르겠다"를 허용)

```
💬 "혈당이라고 하셨는데, 어떤 걸 말씀하시는 걸까요?"
   [ 평소에 재시는 혈당 ]
   [ 병원에서 3개월마다 재는 수치 (당화혈색소) ]
   [ 잘 모르겠어요 ]
```

- **"잘 모르겠어요"** 선택 시 → **추측하지 않고** 일반 정보 + **의료진 상담 권고**로 처리한다.
  고령 참여자가 두 지표를 구분하지 못하는 것은 **정상이며, 그 자체가 연구 관측치**다(아래).

### 연구적 의미 — 이것은 서비스 기능이 아니라 **관측 장치**다

되묻기 기록은 다음 연구 질문에 답한다.

- 고령 당뇨 환자는 자신의 건강 상태를 **어떤 용어로 표현하는가?**
- **혈당과 당화혈색소를 구분하는가?** (구분 못 하는 비율은?)
- AI가 되물었을 때 **답할 수 있는가, "모르겠다"고 하는가?**

→ **AI 건강문해력(AI Health Literacy)의 직접 관측치**다. 설문보다 강력한 증거이며, **설계에 넣지 않으면 영영 관측할 수 없다.**
기록: `messages.content`(원문 발화) · `retrieval_logs.query_text` · `clarification_asked` / `clarification_answered` 이벤트

## 4. 답변 규칙

- **검색된 근거를 우선** 사용한다.
- **근거에 없는 내용을 추측하지 않는다.**
- **답변과 출처를 함께** 표시한다.
- **진단·처방·복약 변경을 제공하지 않는다**(SAFETY_RULES).
- 고령친화적으로 **짧고 쉽게** 답한다.
- 이전 대화 전체가 아니라 **필요한 최근 맥락만** LLM에 전달한다(비용·정확성).

## 5. RAG의 한계 인식

RAG는 할루시네이션을 **줄이는** 장치이지 완전히 제거하지 못한다. 다음 장치를 함께 둔다.

- 승인 문서만 지식베이스에 포함
- 근거 부족 시 보류(단, 3단계로 남발 방지)
- 답변에 사용된 **출처 저장**(`selected_chunk_ids`)
- 안전 질문과 일반 질문 분리
- 전문가가 만든 **테스트 질문으로 검증**(Phase 4)

## 6. 품질 평가 — 검색과 답변을 분리

- **검색 품질**(관련 청크를 잘 찾았는가)과 **답변 품질**(근거를 잘 반영했는가)을 **분리 평가**한다.
- `retrieval_logs`에 검색된 청크·유사도·선택 청크를 남겨, 오답이 **검색 실패인지 생성 실패인지** 구분한다.

## 7. 미결정 사항 (연구팀 확정 — Day 5 정리)

- 최초 지식베이스 승인 문서 목록
- 청킹 의미 단위
- 근거 충분·부족 판단 임계값(상·하한), top-k
- 사용할 LLM·임베딩 모델

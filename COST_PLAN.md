## 2. 비용이 발생하는 4영역

| 영역 | 발생 시점 | 계산 |
|------|-----------|------|
| **문서 임베딩** | 문서 최초 등록, 청킹/모델 변경 시 | 전체 문서 토큰 ÷ 1,000,000 × 임베딩 단가 |
| **질문 임베딩** | 사용자 질문마다 | 질문 토큰 ÷ 1,000,000 × 임베딩 단가 |
| **LLM 답변 생성** | 질문마다 | 입력 토큰 비용 + 출력 토큰 비용 |
| **외부 부가서비스** | 상시/변동 | DB·배포·(향후 알림 등) 별도 |

- LLM 입력 토큰 = 질문 + 시스템 프롬프트 + 검색 청크 + 최근 맥락. 출력 토큰 = 생성된 답변.

## 3. 사전 비용 추정식

```
예상 총 상호작용 수 = 참여자 수 × 연구일 수 × 1일 평균 상호작용 수
예상 LLM 비용      = 총 상호작용 수 × 1회 평균 (입력·출력 토큰 비용)
예상 질문 임베딩   = 총 질문 수 × 1회 평균 질문 토큰 비용
예상 문서 임베딩   = 전체 승인 문서 토큰 수 × 임베딩 단가
```

> 인터랙티브 계산기(cost-calculator 위젯)에서 참여자 수·질문 수·토큰·단가를 넣으면 USD·원화로 자동 추정.

## 4. 비용을 줄이는 기본 원칙

- 모든 넛지에 LLM을 호출하지 않고 **승인 템플릿 우선** 사용(D6).
- RAG 검색 **top-k를 과도하게 늘리지 않는다.**
- 이전 대화 전체가 아니라 **필요한 최근 맥락만** 전달.
- 고령친화적으로 **답변 길이를 짧게** 유지.
- **변경된 문서 청크만 재임베딩**(`is_active`/버전 관리).
- 개발 단계와 실증 단계 모델을 구분.
- 모든 호출의 토큰·비용을 `model_calls`에 저장.

## 5. 기록 필드 (model_calls)

`call_type` · `provider` · `model_name` · `input_tokens` · `output_tokens` · `unit_input_price` · `unit_output_price` · `estimated_cost` · `latency_ms` · `status` · `called_at`

> 가격은 변하므로 **토큰 수뿐 아니라 계산에 쓴 단가·기준일**도 함께 저장한다.

## 6. 비용표에 함께 기록할 것 / 상한

- 모델명 · 단가 확인일 · 환율 · **예비비**
- **월별 비용 상한 기준** _(미결정 — 연구 예산에 맞춰 확정)_. 상한 접근 시 알림·점검.
- 실제 비용은 `model_calls`를 집계해 주기적으로 추정치와 대조.
